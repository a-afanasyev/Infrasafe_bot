# E2E Test Plan — Telegram Bot (@infrasafebot)

> Дата: 2026-03-22 | Ревизия: 3

---

## 1. Инструментарий и ограничения

### 1.1 Доступные MCP-команды (telegram-qa)

| Команда | Назначение |
| ------- | ---------- |
| `tg_connect` | Подключение к Telegram-аккаунту |
| `tg_send_message` | Отправка текстового сообщения |
| `tg_send_command` | Отправка команды (/start, /help и т.д.) |
| `tg_send_contact` | Отправка контакта (шаринг телефона) |
| `tg_click_reply_button` | Нажатие reply-кнопки по тексту |
| `tg_click_inline_button` | Нажатие inline-кнопки по тексту |
| `tg_get_messages` | Получение последних сообщений |
| `tg_get_keyboard` | Получение текущей клавиатуры |
| `tg_wait_response` | Ожидание ответа бота |

### 1.2 НЕ поддерживается MCP telegram-qa

- Отправка фото, видео, документов, стикеров, голосовых, файлов
- Одновременная работа с несколькими Telegram-аккаунтами (только 1 сессия)
- Проверка получения push-уведомлений другим пользователем
- Форвард/реплай на конкретное сообщение

### 1.3 Известный баг MCP

`tg_click_inline_button` возвращает `'BotCallbackAnswer' object has no attribute 'id'`, но клик проходит. Верификация всегда через `tg_get_messages` после клика.

### 1.4 Уровни исполнения

| Метка | Значение | Требования |
| ----- | -------- | ---------- |
| `AUTO` | Полностью автоматизируемый одним MCP telegram-qa | Только текст, кнопки, контакт. Один аккаунт. |
| `HYBRID` | Два драйвера параллельно | MCP telegram-qa + Playwright (web `localhost:5173`) |
| `MANUAL` | Не автоматизируемый текущим MCP | Ручной Telegram-клиент: media, multi-account, blocked user |
| `MULTI_SESSION` | Требует 2+ Telegram-аккаунтов | Верификация уведомлений другому пользователю |

### 1.5 Known gaps (не тестировать)

| Функция | Причина | Файл / строка |
| ------- | ------- | -------------- |
| Управление шаблонами смен (`template_management`) | Ответ бота: "Функция в разработке" | `shift_management.py:2023`, `ru.json:6431` |

## 2. Предусловия и тестовые данные

### 2.1 Telegram-сессии

| # | Аккаунт | Роли | Назначение |
| - | ------- | ---- | ---------- |
| S1 | Основной тестовый | `applicant`, `executor`, `manager` | Все AUTO-тесты, cross-role через "Сменить роль" |
| S2 | Второй тестовый | `applicant` | Верификация уведомлений заявителю (MULTI_SESSION) |
| S3 | Третий тестовый | `executor` | Верификация уведомлений исполнителю, конкурентный доступ (MULTI_SESSION/MANUAL) |

> AUTO-тесты используют только S1. MULTI_SESSION и MANUAL тесты требуют S2/S3.
> MCP telegram-qa подключается только к одному аккаунту за раз — переключение через `tg_disconnect` / `tg_connect`.

### 2.2 Seed data (обязательно перед запуском)

| # | Ресурс | Конкретные требования | Как подготовить |
| - | ------ | --------------------- | --------------- |
| D1 | Справочник адресов | 1 двор ("Yangi Olmazor") → 1 здание ("14V") → квартиры 1-10 | `seed_data.py` или менеджер через бот |
| D2 | S1 основной аккаунт | Зарегистрирован, `approved`, роли: `[applicant, executor, manager]`, квартира привязана (primary) | `/join` + одобрение + онбординг |
| D3 | S2 второй аккаунт | Зарегистрирован, `approved`, роль: `applicant`, квартира привязана | `/join` + одобрение |
| D4 | S3 третий аккаунт | Зарегистрирован, `approved`, роль: `executor`, специализация назначена | `/join` + одобрение |
| D5 | Инвайт-коды | 1 валидный, 1 истёкший, 1 невалидный | Создание через менеджера (раздел 9.3) или БД |
| D6 | Заявки в статусах | Минимум: 2 "Новая", 1 "В работе", 1 "Уточнение" (с вопросом), 1 "Выполнена", 1 "Исполнено", 1 "Отменена" | Создание через S1 + смена статусов через web-панель |
| D7 | Смена на сегодня | 1 запланированная смена для S1 (executor), назначена на текущий день | Web-панель "Смены" или `seed_data.py` |
| D8 | Пользователь на верификацию | 1 pending-пользователь с загруженными документами | Регистрация без одобрения |

### 2.3 Порядок ролевых сессий (cross-role tests, раздел 8)

```text
1. S1 роль: applicant → создать заявку
2. S1 → 🔀 Выбрать роль → manager → назначить исполнителя
3. S1 → 🔀 Выбрать роль → executor → выполнить заявку
4. S1 → 🔀 Выбрать роль → manager → подтвердить выполнение
5. S1 → 🔀 Выбрать роль → applicant → принять заявку
```

Между переключениями: `tg_click_reply_button("🔀 Выбрать роль")` → `tg_click_inline_button(target_role)`.

## 13. Уведомления (MULTI_SESSION)

## Приложение A: Стратегия прогонов

### Слой 1: Automated Smoke (AUTO, P0 only, S1)

Минимальная проверка что бот жив и основные flow работают. Один аккаунт, роль applicant. Без переключения ролей.

1. `/start` → главное меню
2. "📝 Создать заявку" → Сантехника → Квартира → описание → Обычная → пропустить медиа → подтвердить
3. "📋 Мои заявки" → заявка видна, формат корректный (нет `::`)
4. Фильтры: "Все заявки" → "Активные" → "Архив"
5. `/help` → справка без несуществующих команд
6. Профиль → смена языка UZ → проверка → обратно RU

### Слой 2: Automated Regression (AUTO, P0 + P1, S1)

Покрытие основных бизнес-сценариев. Требует seed data (D1-D7). Включает переключение ролей.

1. Все кейсы Слоя 1
2. Создание заявок: все 8 категорий, все 4 срочности, все 3 типа адреса (без медиа)
3. Cross-role happy-path (раздел 8.1): создание → назначение → выполнение → приёмка
4. Уточнение: вопрос → ответ (раздел 8.2)
5. Все кнопки главного меню для 3 ролей (разделы 4.1-4.4)
6. Профиль: телефон, имя, язык (раздел 7)
7. i18n: полный цикл RU ↔ UZ (раздел 9)
8. Панель менеджера: списки пользователей, сотрудников, инвайты (раздел 11)

### Слой 3: MULTI_SESSION (уведомления, S1 + S2 + S3)

Проверка push-уведомлений между аккаунтами. Требует 3 Telegram-сессии.

1. Все кейсы раздела 13
2. Переподключение MCP: `tg_disconnect` → `tg_connect(S2)` → `tg_get_messages` → проверка уведомления

### Слой 4: HYBRID (web ↔ bot, S1 + Playwright)

Выполняется отдельно от bot-only тестов. Требует запущенную web-панель.

1. Все кейсы раздела 15

### Слой 5: Manual Exploratory

Тесты, не автоматизируемые текущим MCP. Выполняются вручную.

1. Отправка фото/видео к заявке (5.4.2, 5.5.5)
2. Загрузка документа при онбординге (3.5.2)
3. Негативные media-кейсы: стикер, голосовое, файл (14.1.3-14.1.5)
4. Конкурентный доступ: два менеджера (14.2.1-14.2.2)
5. Пользователь с одной ролью — кнопка "Сменить роль" не показана (16.5)
