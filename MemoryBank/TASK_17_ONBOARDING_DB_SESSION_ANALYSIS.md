## 🔴 ИСХОДНАЯ ПРОБЛЕМА

**Описание**: 
На строках 177-185 в функции `process_manual_phone()` создавался новый сеанс БД, который затем закрывался. Но на строке 198 использовался уже закрытый сеанс вместо переданного параметра `db`.

**Код с проблемой**:
```python
# Валидируем телефон
phone_number = message.text.strip()
# Получаем язык пользователя для локализации сообщений об ошибках
from uk_management_bot.utils.helpers import get_user_language
from uk_management_bot.database.session import get_db
db = next(get_db())  # ❌ Создается НОВЫЙ сеанс
try:
    user_lang = get_user_language(message.from_user.id, db) or "ru"
except:
    user_lang = "ru"
finally:
    db.close()  # ❌ Закрывается новый сеанс

# ... позже ...
auth_service = AuthService(db)  # ❌ Используется уже закрытый сеанс!
```

**Проблемы**:
1. Функция уже получает параметр `db: Session` (строка 153), но он игнорируется
2. Создается новый сеанс БД, который затем закрывается
3. После закрытия сеанса, параметр `db` (который был переопределен) становится закрытым сеансом
4. При попытке использовать `AuthService(db)` на строке 198 возникает ошибка

---

## 🤔 ПОПЫТКА ИСПРАВЛЕНИЯ

**Мое решение**:
Я удалил создание нового сеанса и попытался использовать существующий параметр `db`:

```python
# Валидируем телефон
phone_number = message.text.strip()
# Получаем язык пользователя для локализации сообщений об ошибках
from uk_management_bot.utils.helpers import get_user_language
try:
    user_lang = get_user_language(message.from_user.id, db) or "ru"
except:
    user_lang = "ru"

is_valid, error_message = Validator.validate_phone(phone_number, language=user_lang)
```

**Проблема с моим решением**:
1. Я предположил, что `Validator.validate_phone()` имеет параметр `language`
2. Но на самом деле метод `validate_phone()` еще не был локализован и не имеет этого параметра (строка 13 в `validators.py`)
3. Согласно плану локализации, валидаторы должны были быть локализованы в Этапе 3, но это еще не было сделано

### Особый случай: validate_urgency()

Метод `validate_urgency()` имеет дополнительную проблему - он использует хардкодные русские строки:

```python
valid_urgencies = ["Обычная", "Средняя", "Срочная", "Критическая"]
```

В то время как в `keyboards/requests.py` уже есть система локализации через `URGENCY_KEYS` и `get_urgency_display()`.

**Последствия**:
- Если пользователь выбирает срочность на узбекском языке, валидация может провалиться
- Несоответствие между отображаемыми текстами (локализованными) и проверяемыми значениями (русскими строками)

**Рекомендация**: 
1. Использовать внутренние ключи из `URGENCY_KEYS.keys()` вместо хардкодного списка
2. Добавить параметр `language` для локализованных сообщений об ошибках
3. Использовать `get_text()` для всех сообщений валидации
