# Руководство системного администратора (system_admin) — UK Management

> Роль: `system_admin`. Хранится в `user.roles` (JSON-массив), активная роль —
> `user.active_role`. Каноническая роль модуля контроля доступа
> (`frontend/src/constants/roles.ts:9-31`). Системный администратор — это менеджер
> с расширенными правами: он видит всё, что видит менеджер, плюс управление
> оборудованием контроля доступа, device-ключами и служебными функциями.

Этот документ описывает **отличия** `system_admin` от `manager`. Общие разделы
дашборда и бота (заявки, аналитика, персонал, адреса, склад, приёмка за заявителя)
описаны в `docs/guides/USER_GUIDE_MANAGER.md` и здесь не повторяются.

## 1. Роль и доступы

### 1.1. Где определяется доступ

- Наборы ролей веб-модулей: `frontend/src/constants/roles.ts`.
  - `ACCESS_MODULE_ROLES` = manager, system_admin, security_operator (строки 47-51) —
    доступ к модулю контроля доступа.
  - `ACCESS_MANAGER_ROLES` = manager, system_admin (строки 59-62) — экраны истории,
    базы доступа и оборудования.
  - `MATERIALS_MODULE_ROLES` = manager, system_admin (строки 69-72) — склад.
- Route guards и вложенные проверки — `frontend/src/App.tsx:119-143`.
- Видимость пунктов меню — `frontend/src/layouts/DashboardLayout.tsx:88-105`
  (группа «Контроль доступа»).

### 1.2. Отличие system_admin от manager (сводно)

| Область | manager | system_admin |
|---|---|---|
| Заявки, аналитика, персонал, адреса, склад, обратная связь, табло | да | да |
| Контроль доступа: обзор, история проездов, база доступа | да | да |
| Контроль доступа: зоны, въезды | да | да |
| Контроль доступа: **камеры, шлагбаумы, edge-контроллеры** | **нет** | **да** |
| Контроль доступа: **device-ключи (выдача/ротация)** | **нет** | **да** |
| Верификация пользователей, роли, права (бот) | да | да |
| Пост охраны (live, ручные открытия) | нет (это `security_operator`) | нет |

Источник разграничения оборудования: `frontend/src/pages/access/AccessEquipmentPage.tsx:62-71`
— бэкенд отдаёт GET по камерам/шлагбаумам/контроллерам только `system_admin`, поэтому
для менеджера эти табы скрыты (доп. гейтинг через `useHasRole`).

## 2. Верификация пользователей (бот)

Полное пошаговое руководство: `uk_management_bot/VERIFICATION_SYSTEM_ADMIN_GUIDE.md`.
Доступ к системе верификации имеют роли **admin** и **manager**
(`VERIFICATION_SYSTEM_ADMIN_GUIDE.md`, раздел «Права доступа»). Ниже — процесс.

Вход: бот → **Панель администратора → Управление пользователями → Панель верификации
пользователей**. Функции панели верификации:

1. **Статистика** — ожидают/верифицированы/отклонены, документы на проверке.
2. **Список пользователей на верификацию** — выбор пользователя, просмотр карточки
   (основные данные, статус, адрес, документы, текущие права).
3. **Запрос дополнительной информации** — типы: точный адрес, паспорт, свидетельство о
   собственности, договор аренды, квитанция ЖКХ, другое; с комментарием.
4. **Проверка документов** — одобрить/отклонить документ, при отклонении обязательна
   причина.
5. **Управление правами на подачу заявок** — уровни: **Квартира** (макс. 2 заявителя),
   **Дом**, **Двор**; предоставление/отзыв прав с комментарием.
6. **Верификация** — одобрить/отклонить пользователя (при отклонении — причина).

Все действия сопровождаются автоматическими уведомлениями пользователю
(`VERIFICATION_SYSTEM_ADMIN_GUIDE.md`, раздел «Уведомления»).

> Верификацию пользователей в боте гейтит `admin`/`manager`
> (`handlers/user_verification.py`). `admin` и `system_admin` — **разные роли**
> (см. [../tech/ROLES_AND_ACCESS.md](../tech/ROLES_AND_ACCESS.md)): `admin` — общий
> управленческий администратор (≈ manager), `system_admin` — админ модуля контроля
> доступа. Полная модель ролей и матрица доступа — в ROLES_AND_ACCESS.md.

## 3. Управление ролями и специализациями (бот)

Файл: `uk_management_bot/handlers/user_management/roles_specs.py`.

- **Роли пользователя** (`roles_specs.py:30-289`): добавление/снятие ролей
  (`role_add_*` / `role_remove_*`), обязательный комментарий к изменению, запрет снятия
  последней роли (`roles_specs.py:131-136`). Сохранение через
  `AuthService.assign_role` / `remove_role` (`roles_specs.py:251-256`).
- **Специализации исполнителя** (`roles_specs.py:294-508`): переключение
  (`spec_toggle_*`), комментарий, сохранение через
  `SpecializationService.set_user_specializations`.
- Проверка прав на управление — `has_admin_access` (`roles_specs.py:37`, `301`).

Приглашения новых пользователей (в т.ч. роли **обходчик**): бот → **Создать
приглашение** (`uk_management_bot/keyboards/admin.py:79-116`) — выбор роли, для
исполнителя специализации, срок действия ссылки.

## 4. Оборудование контроля доступа (только system_admin)

Экран **Контроль доступа → Оборудование** — `/dashboard/access/equipment`
(`frontend/src/pages/access/AccessEquipmentPage.tsx`). Route защищён
`ACCESS_MANAGER_ROLES` (`App.tsx:135-137`); внутри табы фильтруются по роли
(`AccessEquipmentPage.tsx:62-71`).

Табы (`AccessEquipmentPage.tsx:71`):

- **Зоны** (`zones`) — manager+system_admin. Код, название, тип парковки
  (закреплённая/общая, `ParkingTypeBadge`), ёмкость, лимит авто на квартиру,
  привязанные фазы (yards). Для общих зон — занятость (`ZoneOccupancyCell`,
  `AccessEquipmentPage.tsx:84-90`).
- **Места** (`spots`) и **Закрепления** (`assignments`) — места и их закрепление за
  квартирами (владение/аренда со сроком, отзыв/продление; хуки `useParkingAdmin`).
- **Въезды** (`gates`) — manager+system_admin. Точка проезда и направление.
- **Камеры** (`cameras`) — **только system_admin**. Привязка к въезду, вендор/модель,
  доп-атрибуты (JSON).
- **Шлагбаумы** (`barriers`) — **только system_admin**. Тип реле, конфиг (JSON).
- **Контроллеры (edge)** (`controllers`) — **только system_admin**. Создание
  контроллера выдаёт **device API-ключ один раз** (`ControllerKeyDialog`) — сохранить
  сразу; есть **ротация ключа** (`useRotateControllerKey`,
  `AccessEquipmentPage.tsx:41`).
- **Диагностика точки** без камеры — «Тест» (`ControllerTestDialog`): синтетический
  проезд через реальный Decision Engine, видно решение/команду; событие помечается как
  диагностическое.

Подробные сценарии оборудования (зоны с местами, аренда/владение, приёмка точки без
камеры) — в `docs/access-control/guides/manager.md`, раздел 3.

> Пост охраны (`security_operator`) — отдельная роль и интерфейс, см.
> `docs/access-control/guides/operator.md`. `system_admin` оборудование настраивает,
> но на посту охраны не работает.

Референсы модуля: `docs/access-control/TECHNICAL_SPEC.md`,
`docs/access-control/DATA_MODEL_PILOT.md`, `docs/access-control/DEPLOYMENT_CHECKLIST.md`.

## Открытые вопросы (проверить у владельца продукта)

- ✅ **РЕШЕНО (2026-07-06):** `admin`, `manager`, `system_admin` — **три разные роли**.
  `admin` — общий управленческий администратор (≈ manager; ~116 authz-сайтов бота+API,
  `App.tsx:102`), но **не в каноническом реестре** `USER_ROLES` (код-долг). `system_admin` —
  админ модуля контроля доступа (оборудование, `AccessEquipmentPage.tsx:914`). `admin` в
  RBAC access-control НЕ участвует. Полная модель — [../tech/ROLES_AND_ACCESS.md](../tech/ROLES_AND_ACCESS.md).
- ⚠️ **[ТРЕБУЕТ УТОЧНЕНИЯ]** Может ли `system_admin` работать на посту охраны или это
  строго `security_operator`? В коде посты охраны гейтятся `security_operator`
  (`docs/access-control/guides/operator.md`), но пересечение полномочий не описано.
