## Critical

### BOT-3: Executors can't start shifts — legacy `user.role` check

- **File**: `uk_management_bot/services/shift_service.py:75`
- **Description**: `start_shift()` checks `user.role` (legacy single-value field) instead of `user.active_role` or `user.roles`. For users with `user.role = "applicant"` but `user.roles = ["applicant","executor","manager"]` and `user.active_role = "executor"`, the shift start is denied with "Доступ запрещен".
- **Impact**: All executors who were originally registered as applicants cannot start shifts.
- **Fix**: Replace `user.role` check with `user.active_role` or parse `user.roles`.

```python
# shift_service.py:75 — current (broken)
if user.role not in [ROLE_EXECUTOR, ROLE_MANAGER]:
    return {"success": False, "message": "Доступ запрещен", "shift": None}

# fix
from uk_management_bot.utils.auth_helpers import get_user_roles
user_roles = get_user_roles(user)
if not any(r in [ROLE_EXECUTOR, ROLE_MANAGER] for r in user_roles):
    return {"success": False, "message": "Доступ запрещен", "shift": None}
```

### BOT-4: `@require_role` fails for callback query handlers

- **File**: `uk_management_bot/middlewares/auth.py:174`
- **Description**: `@require_role` decorator looks for `roles` in handler kwargs, but for callback query handlers aiogram DI does not inject `roles` — only `language` and `state` are present in kwargs. The decorator falls through to the DB lookup path, but `db` is also missing from kwargs for these handlers.
- **Log**: `require_role: user_roles пустой! kwargs=['language', 'state'], event=CallbackQuery, telegram_id=6055402868`
- **Impact**: Any callback handler decorated with `@require_role` denies access to all users. Affects: "Мои смены" sub-menus (Расписание, История, Статистика, etc.).
- **Fix**: The decorator must always fall back to DB lookup when `roles` kwarg is missing. Ensure `db` is obtained either from kwargs or via `SessionLocal()`.

```python
# In require_role wrapper, after getting user_roles from kwargs:
if not user_roles and telegram_id:
    # Always try DB fallback, even without db in kwargs
    if not db:
        from uk_management_bot.database.session import SessionLocal
        db = SessionLocal()
        need_close = True
    try:
        from uk_management_bot.database.models.user import User as _User
        user = db.query(_User).filter(_User.telegram_id == telegram_id).first()
        if user:
            from uk_management_bot.utils.auth_helpers import get_user_roles as _get_user_roles
            user_roles = _get_user_roles(user)
    finally:
        if need_close:
            db.close()
```

## High

### WEB-3: WebSocket 403 — real-time kanban updates broken

- **File**: nginx config (frontend proxy)
- **Description**: WebSocket connection to `ws://localhost:3002/ws/v2/kanban?token=...` fails with 403 during handshake. The nginx reverse proxy is not forwarding WebSocket upgrade headers.
- **Impact**: Kanban board does not receive real-time updates; requires manual page refresh.
- **Fix**: Add WebSocket upgrade headers to nginx proxy config:

```nginx
location /ws/ {
    proxy_pass http://api:8085;
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "upgrade";
}
```
