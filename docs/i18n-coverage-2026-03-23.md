## Найденные проблемы перевода

### I18N-1 (Critical): /start всегда показывает русский

- **Причина**: Известный BOT-1 — `base.py:145` не передаёт `language` в `handle_regular_start`
- **Затронуто**: Приветственное сообщение + вся клавиатура после /start
- **Масштаб**: Каждый раз при /start узбекоязычный пользователь видит русский текст. При переключении роли клавиатура корректно показывается на UZ.

### I18N-2 (Low): Статусы заявок не переведены в "Мои заявки"

- **Пример**: "#260323-007 - Santexnika - **Новая**" — "Новая" на русском
- **Причина**: Статусы хранятся в БД на русском ("Новая", "В работе", "Выполнена" и т.д.) и выводятся as-is без маппинга на текущий язык
- **Затронуто**: Список "Мои заявки", все страницы
- **Fix**: Добавить маппинг `status → localized_status` при отображении, например:
  ```python
  STATUS_KEYS = {
      "Новая": "statuses.new",
      "В работе": "statuses.in_progress",
      "Выполнена": "statuses.completed",
      ...
  }
  localized = get_text(STATUS_KEYS.get(status, status), language=lang)
  ```

### I18N-3 (Low): Префиксы адресов не переведены

- **Пример**: "Двор: Фаза 1(LOT 4)", "Дом: Yangi Olmazor", "Квартира 54"
- **Причина**: Префиксы "Двор:", "Дом:", "Квартира" формируются на русском в коде или хранятся в БД
- **Затронуто**: Списки заявок, сводка заявки, профиль
- **Масштаб**: Косметическая проблема, адреса узнаваемы

### I18N-4 (Info): "кв." в профиле

- **Пример**: "Yangi Olmazor, 14V, кв. 54" — "кв." на русском
- **Причина**: Формат адреса в БД содержит русское сокращение
- **Масштаб**: Незначительно

## План исправления

### FIX-1: `/start`, `/menu` и «Перезапуск» игнорируют язык пользователя (I18N-1 / BOT-1)

**Приоритет**: Critical

**Проблема состоит из двух независимых частей:**

#### A. `language` не передаётся в `handle_regular_start`

Три вызова `handle_regular_start` не передают `language`:

| Вызов | Файл | Строка |
| ----- | ---- | ------ |
| `/start` | `handlers/base.py` | 145 |
| `/menu` | `handlers/base.py` | 237 |

Оба хэндлера получают `language` через DI (из `localization_middleware`), но вызывают `handle_regular_start(message, db, roles, active_role, user_status)` без него. Функция использует `language="ru"` по умолчанию.

**Исправление** — добавить `language=language` в оба вызова:

```python
# base.py:145
await handle_regular_start(message, db, roles, active_role, user_status, language=language)

# base.py:237
await handle_regular_start(message, db, roles, active_role, user_status, language=language)
```

#### B. Язык перезаписывается из Telegram-клиента при каждом `/start` и «Перезапуск»

Три точки перезаписи `user.language` в БД значением `from_user.language_code`:

| Точка | Файл | Строка | Когда срабатывает |
| ----- | ---- | ------ | ----------------- |
| `handle_regular_start` | `handlers/base.py` | 160-163 | Каждый `/start` и `/menu` |
| `handle_restart_bot` | `handlers/base.py` | 256-260 | Нажатие inline-кнопки «Перезапуск» |

Сценарий поломки: пользователь выбирает UZ через меню профиля (`profile_editing.py:220` сохраняет `user.language = "uz"`). Затем нажимает `/start` — `handle_regular_start:160` перезаписывает `user.language` на `message.from_user.language_code` (язык Telegram-клиента, обычно `"ru"`). Следующий запрос через `localization_middleware` получает `"ru"` из БД.

Защита `auth_service.update_user_language` проверяет `SUPPORTED_LANGUAGES`, но не различает автоопределение от ручного выбора.

**Исправление** — убрать автоперезапись для существующих пользователей. Автоопределение нужно только при первом `/start` (когда `user.language` ещё не установлен):

```python
# base.py:160-163 — автоопределение только для новых пользователей
if message.from_user.language_code and not user.language:
    await auth_service.update_user_language(
        message.from_user.id,
        message.from_user.language_code
    )

# base.py:256-260 — удалить блок перезаписи полностью:
# (callback «Перезапуск» не должен менять язык)
# УДАЛИТЬ:
# if callback.from_user.language_code:
#     await auth_service.update_user_language(
#         callback.from_user.id,
#         callback.from_user.language_code
#     )
```

**Примечание**: условие `not user.language` безопасно, т.к. все существующие пользователи уже имеют `user.language` заполненным (через онбординг или profile_editing). Для новых пользователей `user.language` будет `None` до первого сохранения.

**Проверка**:

1. Выбрать UZ в профиле → `/start` → приветствие и клавиатура на узбекском
2. `/menu` → меню на узбекском
3. Нажать inline «Перезапуск» → бот на узбекском
4. Перезапустить бота → язык сохранён

### FIX-3: Адресные префиксы не переведены при отображении (I18N-3 + I18N-4)

**Приоритет**: Low

**Проблема**: при создании заявки адрес сохраняется в БД с локализованным префиксом (`get_text("requests.building_prefix")`). Новые заявки на UZ сохраняют `"Uy: Yangi Olmazor"`, но все старые заявки содержат `"Дом: Yangi Olmazor"`. Также `"кв. 54"` не переводится.

**Масштаб проблемы**: `request.address` отображается в ~16 точках кода. Полный список файлов для патча:

**Через helper-функции** (покрывают основные экраны):

1. `utils/request_helpers.py:136` — `format_request_details` (детальный вид заявки)
2. `utils/request_helpers.py:280` — `format_request_list_item` (списки заявок)
3. `utils/request_helpers.py:84-86` — `format_request_summary` (краткая сводка, **не принимает `language`** — нужно добавить параметр или оставить без локализации)

**Прямое использование `request.address`** (каждый — отдельный патч):

4. `handlers/requests.py:1266` — `handle_view_request` (просмотр заявки)
5. `handlers/requests.py:2469` — `show_my_requests` (мои заявки)
6. `handlers/admin.py:316` — детали в админ-панели
7. `handlers/admin.py:2676,2706` — списки в админ-панели
8. `handlers/request_acceptance.py:156` — приёмка
9. `handlers/request_reports.py:365` — отчёты
10. `handlers/unaccepted_requests.py:87,250` — неподтверждённые заявки
11. `handlers/request_assignment.py:199,244` — назначение исполнителя
12. `handlers/clarification_replies.py:61,149` — уточнение по заявке
13. `handlers/request_status_management.py:688` — смена статуса
14. `services/notification_service.py:467` — уведомления
15. `services/shift_transfer_service.py:179` — передача смены
16. `utils/helpers.py:239` — `format_request_info`

**Рекомендуемый подход** — централизованная функция + метод на модели:

Файл: `uk_management_bot/utils/address_helpers.py` (новый)

```python
def localize_address(address: str, language: str) -> str:
    """Заменяет русские адресные префиксы на локализованные при отображении.

    Адреса хранятся в БД в том виде, в котором были созданы (зависит от языка
    пользователя на момент создания). Эта функция нормализует отображение
    старых адресов с русскими префиксами при показе на узбекском.
    """
    if language == "ru" or not address:
        return address

    replacements = [
        ("Дом: ", "Uy: "),
        ("Двор: ", "Hovli: "),
        ("кв. ", "xon. "),
    ]
    for ru_prefix, uz_prefix in replacements:
        if ru_prefix in address:
            address = address.replace(ru_prefix, uz_prefix)
    return address
```

Затем в модели `Request` (`database/models/request.py`) добавить convenience-метод:

```python
def get_localized_address(self, language: str = "ru") -> str:
    from uk_management_bot.utils.address_helpers import localize_address
    return localize_address(self.address, language)
```

Это позволит в каждой из 16 точек заменить `request.address` на `request.get_localized_address(language)` без дополнительных импортов.

**Ограничения подхода**:

- Хардкод замен `("Дом: ", "Uy: ")` — при добавлении третьего языка нужно расширять
- `str.replace` заменяет все вхождения, а не только префикс. Для `"Дом: "` и `"Двор: "` это безопасно (маловероятно появление в середине адреса). Для `"кв. "` теоретически возможно ложное срабатывание, но в контексте адресов `"кв. "` всегда означает квартиру

**Проверка**: переключить на UZ → просмотреть старую заявку → `"Uy: Yangi Olmazor, 14V, xon. 54"`.
