## S1 — Blockers

### TWA-01: Submit-кнопка не работала из-за enum mismatch у category ✅ FIXED (локально)

- **Files**: `frontend/src/twa/pages/applicant/CreatePage.tsx:10-23`, `:44-52`
- **Description**: TWA отправлял `category: 'electricity'`, backend `CreateRequestBody.validate_category` требовал значение из `settings.REQUEST_CATEGORIES = ['Электрика', 'Сантехника', ...]` (`uk_management_bot/config/settings.py:121-132`). Pydantic возвращал 422, `onError` только писал `console.error('Mutation failed:', err)` без UI-feedback.
- **Impact**: ВСЕ заявки из мини-аппа возвращали 422 — функция была сломана полностью. Юзер видел "кнопка ничего не делает".
- **Fix**: добавлен `CATEGORY_API_MAP` (зеркало `URGENCY_API_MAP`) + inline error display под кнопкой submit (показывает `detail` из 422). Локально готово, требует deploy.

### TWA-02: Фото при создании заявки никуда не отправляются

- **Files**: `frontend/src/twa/pages/applicant/CreatePage.tsx:36`, `:43-52`, `:102-109`
- **Description**: На step 3 wizard'а юзер выбирает фото — `setPhotos: File[]` пишет в state. На step 5 submit мутация `createMutation.mutationFn` строит payload `{category, apartment_id, address, description, urgency, source}` — поля `photos` / `media_files` отсутствуют. Файлы остаются только в локальном state.
- **Impact**: Юзер видит превью загруженных фото, нажимает «Отправить» — заявка создаётся, но без фото. Менеджер и исполнитель никогда не видят прикреплённых изображений.
- **Fix**: После успешного POST `/api/v2/requests` → получить `request_number` → for each photo: POST `/api/v1/media/upload` с `request_number`, `category=request_photo` через media-service (или через прокси-эндпоинт на API). Затем patch заявку с `media_files: [id, ...]`. Бэкенд `CreateRequestBody.media_files: Optional[List[str]]` уже принимает (`api/requests/schemas.py:60`), нужен только клиентский flow.

## S2 — Broken features

### TWA-03: Тихое подавление ошибок на всех mutation-страницах (паттерн)

- **Files (occurrences)**:
  - `applicant/AcceptancePage.tsx:34-37`, `:49-52` — accept & return
  - `executor/ProfilePage.tsx:24-27` — смена языка
  - `executor/CompletionReport.tsx:26` — `onError: () => haptic('notification')` (вибрирует, но без объяснения)
  - `executor/TaskDetailPage.tsx` — `statusMutation` без `onError`
- **Description**: Паттерн `onError: (err) => console.error('Mutation failed:', err)` повторяется в 6+ местах. Юзер не получает никакой обратной связи (422, 403, network) — кнопка нажимается, ничего не происходит. После фикса CATEGORY mapping это снова приведёт к запутыванию при следующей ошибке.
- **Impact**: Любая 4xx/5xx → юзер думает, что приложение зависло. Заведомо плохой UX, маскирует баги.
- **Fix**: централизовать через toast (`sonner` или похожая компактная либа) или унифицированный inline-error блок (как уже сделан в CreatePage step 5).

### TWA-04: CompletionReport не инвалидирует кэш detail-страницы

- **File**: `frontend/src/twa/pages/executor/CompletionReport.tsx:21-25`
- **Description**: После PATCH `/api/v2/requests/{number}` с status='Выполнена', `onSuccess` инвалидирует только `['executor-tasks']`. Кэш `['request', number]` (TaskDetailPage) остаётся с прежним status='В работе'. После navigate('/twa/exec') юзер может вернуться назад → видит устаревшие данные.
- **Impact**: UI рассинхрон. Стабильный сценарий с stale state.
- **Fix**: добавить `queryClient.invalidateQueries({ queryKey: ['request', number] })` в `onSuccess`.

### TWA-05: useTWAAuth — гонка cold-start refresh vs Telegram initData

- **File**: `frontend/src/twa/hooks/useTWAAuth.ts` (целиком)
- **Description**: На холодном старте `authenticate` callback может запуститься до того, как Telegram WebApp отдаст `initData` (SDK не использует промис `ready()`). Тогда фоллбек идёт на localStorage refresh. Когда initData всё-таки прилетит — callback пересоздастся (deps array) и effect пере-запустит логику. Возможен лишний refresh-запрос. В edge-кейсе с очень медленной инициализацией Telegram — первый запрос пользователя падает с 401 до того, как initData дошёл.
- **Impact**: Лишние refresh-запросы (rate-limit риск), потенциальный 401 на первом API-вызове после открытия.
- **Fix**: ждать `tg.ready()` (или поллить `tg?.initData` до non-empty) перед первым `authenticate`. Добавить guard `if (accessToken) return` в effect.

### TWA-06: 401 → refresh fails → тихий auth dead-end

- **File**: `frontend/src/twa/twaClient.ts:34-45`
- **Description**: Если `/api/v2/auth/refresh` возвращает 401 (refresh-token отозван), catch чистит `localStorage.twa_refresh_token` и резолвит `''`. Дальше `if (newToken)` ложно — оригинальный запрос rejected. Но **никакого сигнала App-у об auth failure**. Все последующие запросы пойдут уже с пустым/старым `Authorization` header — серия молчаливых 401. Нет редиректа на login и нет UI о "сессия истекла".
- **Impact**: После истечения refresh-token (или revoke) — мини-апп навечно "сломан" для юзера до полного reload Telegram.
- **Fix**: при refresh 401 — эмитить событие (CustomEvent на window или Zustand state) → App.tsx показывает экран "сессия истекла, перезапустите приложение" или повторно вызывает `authenticate(initData)`.

### TWA-07: Concurrent Accept/Return на AcceptancePage

- **File**: `frontend/src/twa/pages/applicant/AcceptancePage.tsx:88-101`
- **Description**: Кнопка Accept disabled только при `acceptMutation.isPending`. Кнопка Return — только при `returnMutation.isPending`. Юзер может нажать Accept, и пока в полёте — нажать Return на той же заявке. Бэкенд получает обе PATCH-операции с разными статусами одновременно.
- **Impact**: Race condition; результирующий статус заявки непредсказуем. Возможен 409/конфликт без UI-обратной связи (см. TWA-03).
- **Fix**: обоим кнопкам `disabled = acceptMutation.isPending || returnMutation.isPending`.

### TWA-08: initData не верифицируется при refresh-токене

- **File**: `uk_management_bot/api/auth/router.py` (эндпоинт `/api/v2/auth/refresh`)
- **Description**: Первичный TWA-логин валидирует hash initData и auth_date ≤5 мин. Refresh-эндпоинт верифицирует только сам refresh-JWT — никакой повторной привязки к initData. Если refresh-token leak-нулся (через XSS, lost device, etc.), он валиден 30 дней независимо от того, отозвал ли пользователь Telegram-сессию.
- **Impact**: Compromised/revoked Telegram session → доступ к API сохраняется до 30 дней.
- **Fix** (один из вариантов):
  - сократить refresh TTL до 15 мин + при истечении заставлять клиент сделать `/twa-init` заново;
  - либо хранить `initData_hash` в `RefreshToken` модели и при `/refresh` периодически (каждые N часов) требовать свежий initData;
  - либо привязать refresh к user-agent / telegram_id и проверять при refresh.

### TWA-09: Bot — false-success log при сбое media upload

- **Files**: `uk_management_bot/utils/media_helpers.py:210`, `uk_management_bot/handlers/onboarding.py`
- **Description**: `upload_document_to_media_service` ловит exception и возвращает `None` (правильно). Но handler в `onboarding.py` всё равно пишет `logger.info("Документ пользователя X загружен в Media Service")` безусловно, не проверяя результат. (Это был источник путаницы при диагностике 401 на media-service сегодня.)
- **Impact**: Логи показывают success при провалах → маскирует баги, усложняет debugging.
- **Fix**: в `handlers/onboarding.py` проверять `if result is not None` перед success-логом; иначе лог уровня warning/error.

## S3 — UX & Hygiene

### TWA-10: PhotoUploader утечка blob URLs

- **File**: `frontend/src/twa/components/PhotoUploader.tsx:31-37`
- **Description**: `URL.createObjectURL(file)` revoke-ится только в callback `onLoad` тега `<img>`. Если файл удалён до загрузки превью, или компонент размонтирован, или onLoad не сработал — blob URL остаётся в памяти браузера навсегда.
- **Impact**: На длинных сессиях с многократным выбором/удалением фото — постоянно растущее потребление памяти. Не критично, но bad practice.
- **Fix**: `useEffect` с cleanup, который пройдёт по текущему `files` и сделает `URL.revokeObjectURL()` для blob URL при размонтировании или изменении массива.

### TWA-11: BackButton не скрывается при возврате на step 0 в CreatePage

- **File**: `frontend/src/twa/pages/applicant/CreatePage.tsx:28-30`
- **Description**: `useEffect(() => { if (step > 0) return showBackButton(goBack) }, [step, ...])` — при `step === 0` effect не возвращает cleanup. Если BackButton был ранее показан (step > 0), он не скроется при возврате на step 0; и наоборот, если юзер навигирует прочь и обратно — может остаться "висящий" callback от старого render.
- **Impact**: Возможен непрошенный visible BackButton, либо клик по нему вызывает stale closure `goBack`.
- **Fix**: всегда возвращать cleanup из effect; при step=0 — явно `tg?.BackButton?.hide()`.

### TWA-12: Нет client-side role check для executor-маршрутов

- **Files**: `frontend/src/twa/pages/executor/*.tsx` (все), `frontend/src/twa/App.tsx`
- **Description**: Страницы исполнителя (`/twa/exec/*`) предполагают, что пользователь имеет роль executor, но ни одна не проверяет это клиентски. Applicant, открывший `/twa/exec/tasks`, увидит рендер страницы, дальше запросы (`scope=my` под executor контекстом) вернут 403 или пустой список, UI покажет "нет задач".
- **Impact**: Юзер с неправильной ролью видит "пустой кабинет исполнителя" вместо понятного сообщения.
- **Fix**: добавить `<RoleGuard required="executor">` компонент-обёртку в App.tsx; при отсутствии роли — редирект на `/twa/app`.

## S4 — Minor / by-design

### TWA-13: Wizard CreatePage теряет state при сворачивании мини-аппа

- **File**: `frontend/src/twa/pages/applicant/CreatePage.tsx` (state)
- **Description**: Все шаги — react state. При случайном close+reopen Telegram WebApp (например, переключение приложения на телефоне) — весь wizard сбрасывается на step 0, фото теряются.
- **Impact**: Юзер пишет длинное описание, выбирает фото, перепрыгнул в Telegram-чат, вернулся — всё начинать сначала.
- **Fix**: persist wizard state в `sessionStorage` или `localStorage` до момента submit; чистить после success или явного "cancel".

### TWA-14: PullToRefresh не настроен на applicant-страницах

- **Files**: `frontend/src/twa/pages/applicant/RequestsPage.tsx`, `HomePage.tsx`
- **Description**: Executor `TasksPage.tsx:32` оборачивает контент в `<PullToRefresh queryKeys={[['executor-tasks']]}>`. Applicant аналоги не используют этот компонент — обновить список заявок можно только через перезаход в страницу или мутацию.
- **Impact**: Юзер ждёт авто-обновления, его нет, фрустрация.
- **Fix**: обернуть applicant-списки в PullToRefresh с правильными queryKeys.
