## S1 — Security

### TWA-19: IDOR на `/api/v2/media/{media_id}/file`

- **File**: `uk_management_bot/api/main.py:343-378` (`proxy_media_file`)
- **Vote**: 3/3 CONFIRMED
- **Description**: Эндпоинт принимает `media_id: int` и проверяет только аутентификацию через `Depends(get_current_user)`. Авторизация (что именно этот юзер имеет доступ к заявке, к которой относится файл) отсутствует. Поскольку `media_id` — последовательный integer (1, 2, 3, …), перебор тривиален.
- **Impact**: Любой авторизованный TWA-пользователь может скачать любое фото в системе. Confidentiality breach.
- **Fix**: после получения данных файла резолвить `media_file.request_number` → найти заявку → вызвать `check_request_access(user, request)` из `api.dependencies_access`. Альтернатива: проксировать только если пользователь — applicant заявки, назначенный executor, или manager.

## S2 — Broken features

### TWA-20: Telegram BackButton выкидывает со страницы при открытом lightbox

- **Files**: `frontend/src/twa/pages/applicant/RequestDetailPage.tsx:17-19`, `frontend/src/twa/pages/executor/TaskDetailPage.tsx:29-31`, `frontend/src/twa/components/MediaGallery.tsx` (Lightbox)
- **Vote**: 3/3 CONFIRMED
- **Description**: На обеих detail-страницах безусловно регистрируется `showBackButton(() => navigate(-1))`. Lightbox не имеет механизма перехвата BackButton. Когда юзер открыл фото на весь экран и нажал Telegram BackButton → срабатывает `navigate(-1)` со страницы, вместо закрытия модалки.
- **Impact**: Открыл фото → нажал назад → улетел из деталей заявки. Сбивает с толку и теряет контекст.
- **Fix**: MediaGallery exposes lightbox-open state через ref или callback prop; в `useEffect` страницы — handler сначала проверяет, открыт ли lightbox, если да — `closeLightbox()` + `return`. Альтернатива: модалка сама диспатчит свой BackButton handler через `showBackButton` (он pop'нет navigation handler).

### TWA-24: Cascade 401 не эскалирует в `twa:auth-failed`

- **File**: `frontend/src/twa/twaClient.ts:21-50`
- **Vote**: 2/3 CONFIRMED (verifier C ошибочно сослался на line 39 — там dispatch при провале **самого refresh**, не при повторном 401 после успешного refresh + retry)
- **Description**: Interceptor: 401 → refresh-token взят → axios.post `/refresh` → если refresh успешен, retry оригинального запроса. Если retry **тоже** возвращает 401, `_retry=true` уже выставлен на первом проходе → второй прогон через interceptor пропустит refresh-логику и просто `Promise.reject(error)`. **Никаких** диспатчей `twa:auth-failed`. useTWAAuth не узнает, что нужна re-аутентификация.
- **Impact**: Редкий сценарий: server ротировал refresh-token, выдал новый access, но access уже отозван (race condition или revocation). Юзер видит ошибку в toast, но сессия в фронте формально активна → все следующие запросы будут 401-401-… в тишине.
- **Fix**: после await retried request, если он `.response?.status === 401` — диспатчить `window.dispatchEvent(new CustomEvent('twa:auth-failed'))`. Либо в catch-блоке вокруг retry.

### TWA-28: Offline-баннер + toast «Network Error» = двойной шум

- **Files**: `frontend/src/twa/components/OfflineIndicator.tsx` + все mutation onError
- **Vote**: 2/3 CONFIRMED, 1/3 PARTIAL (соглашается: оба сигнала действительно показываются; нюанс — OfflineIndicator сам по себе toast не диспатчит)
- **Description**: Когда `navigator.onLine === false`, OfflineIndicator рисует баннер. Если юзер успевает нажать кнопку до того как реальная сеть восстановится, mutation выйдет → fail → `onError` → `toast.error(getErrorMessage(err))` → "Network Error".
- **Impact**: Юзер видит и постоянный баннер «Нет соединения» и spam'ом toast'ы для каждой попытки. Confusing.
- **Fix**: в `getErrorMessage` или в central wrapper — если `!navigator.onLine`, возвращать пустую строку, тогда toast не показывать. Альтернатива: глобально в `useMutation` defaults подключить guard `meta: { hideErrorWhenOffline: true }`.
