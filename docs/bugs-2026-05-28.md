## S1 — Security

### TWA-19: IDOR на `/api/v2/media/{media_id}/file`

- **File**: `uk_management_bot/api/main.py:343-378` (`proxy_media_file`)
- **Vote**: 3/3 CONFIRMED
- **Description**: Эндпоинт принимает `media_id: int` и проверяет только аутентификацию через `Depends(get_current_user)`. Авторизация (что именно этот юзер имеет доступ к заявке, к которой относится файл) отсутствует. Поскольку `media_id` — последовательный integer (1, 2, 3, …), перебор тривиален.
- **Impact**: Любой авторизованный TWA-пользователь может скачать любое фото в системе. Confidentiality breach.
- **Fix**: после получения данных файла резолвить `media_file.request_number` → найти заявку → вызвать `check_request_access(user, request)` из `api.dependencies_access`. Альтернатива: проксировать только если пользователь — applicant заявки, назначенный executor, или manager.

### TWA-24: Cascade 401 не эскалирует в `twa:auth-failed`

- **File**: `frontend/src/twa/twaClient.ts:21-50`
- **Vote**: 2/3 CONFIRMED (verifier C ошибочно сослался на line 39 — там dispatch при провале **самого refresh**, не при повторном 401 после успешного refresh + retry)
- **Description**: Interceptor: 401 → refresh-token взят → axios.post `/refresh` → если refresh успешен, retry оригинального запроса. Если retry **тоже** возвращает 401, `_retry=true` уже выставлен на первом проходе → второй прогон через interceptor пропустит refresh-логику и просто `Promise.reject(error)`. **Никаких** диспатчей `twa:auth-failed`. useTWAAuth не узнает, что нужна re-аутентификация.
- **Impact**: Редкий сценарий: server ротировал refresh-token, выдал новый access, но access уже отозван (race condition или revocation). Юзер видит ошибку в toast, но сессия в фронте формально активна → все следующие запросы будут 401-401-… в тишине.
- **Fix**: после await retried request, если он `.response?.status === 401` — диспатчить `window.dispatchEvent(new CustomEvent('twa:auth-failed'))`. Либо в catch-блоке вокруг retry.

# Manager-flow audit (Telegram bot) — 2026-05-28

Проверка функционала роли **manager** через telegram-qa MCP на **проде** (тестовый аккаунт id=4, tg_id `6055402868`, временно повышен до manager, по окончании восстановлен в `["executor"]`). Прошёл по: управление жителями, управление сотрудниками, добавление/удаление специализаций.

## MGR-01 (S1) — Любое аудируемое действие менеджера падает для Telegram ID > 2^31 (int4 overflow в `audit_logs.telegram_user_id`)

- **Files**:
  - Прод-схема: `audit_logs.telegram_user_id` имеет тип `integer` (int4, max 2 147 483 647).
  - Модель: `uk_management_bot/database/models/audit.py:16` — `telegram_user_id = Column(Integer, ...)` (должен быть `BigInteger`; `users.telegram_id` уже `bigint`).
  - Существующая, но **не давшая эффекта** миграция: `alembic/versions/004_audit_logs_bigint_telegram_id.py` (alter integer→bigint).
- **Status**: ✅ FIXED & verified on prod (2026-05-28). Воспроизведено вживую (2 сценария), затем исправлено и перепроверено round-trip'ом.
- **Description**: Все хендлеры менеджера пишут запись в `audit_logs` в той же транзакции, что и само изменение. Insert передаёт `telegram_user_id` (Telegram ID актора или цели). Когда это значение выходит за пределы int4 — `psycopg2.errors.NumericValueOutOfRange: integer out of range` → транзакция откатывается → действие **не выполняется**.
  - Современные Telegram ID регулярно превышают 2^31. ID самого тестового менеджера `6055402868` уже выходит за диапазон → менеджер с таким ID не может выполнить **ни одного** аудируемого действия.
- **Воспроизведение**:
  1. *Одобрение жителя* (цель id=2, tg_id `-9999999991`): UI → «✅ Одобрить» → ввод комментария → бот: «❌ Операция не выполнена. Попробуйте позже». Статус остался `pending`.
     - Лог: `auth_service` — `Ошибка одобрения пользователя 2: NumericValueOutOfRange ... telegram_user_id: -9999999991, action: user_approved`.
  2. *Смена специализаций* (цель = сам тестовый менеджер id=4, актор tg_id `6055402868`): UI → «🔧 Специализация» → добавить «⚡ Электрик» → «💾 Сохранить» → ввод комментария → бот: «❌ Произошла ошибка при обновлении специализаций» + «⚠️ Произошла непредвиденная ошибка». `specialization` остался `["plumber"]`.
     - Лог: `handlers.employee_management` — `Ошибка обработки комментария специализаций: NumericValueOutOfRange ... telegram_user_id: 6055402868, action: specialization_change`.
     - Каскад: ошибка во flush → `db.commit()` в `uk_management_bot/main.py:246` (db_middleware) → `PendingRollbackError` → Unhandled exception в aiogram error-middleware → пользователь видит generic «⚠️ Произошла непредвиденная ошибка».
- **Impact**: **S1 / blocker.** Ломает основные функции менеджера для целого класса пользователей. Под удар попадают любые действия, пишущие audit_log: одобрить/заблокировать/разблокировать/отклонить жителя и сотрудника, смена ролей, смена специализаций — если tg_id актора **или** цели > 2^31. Поскольку ID актора-менеджера уже может быть > 2^31, для такого менеджера блокируется ВСЁ. Тихо: данные не меняются, юзер видит общую ошибку.
- **Почему миграция 004 не помогла**: прод `alembic_version = 009` (т.е. формально 004 «применена»), но колонка по факту `integer`. Вероятно таблица создана из модели (`Integer`) через `create_all`, а версии застемплены (`alembic stamp`) без фактического `ALTER`. Повторный `alembic upgrade` ничего не изменит — alembic считает 004 выполненной.
- **Fix**:
  1. Прод (требует подтверждения — это ALTER на проде): `ALTER TABLE audit_logs ALTER COLUMN telegram_user_id TYPE BIGINT;` (не блокирующая операция на Postgres для расширения int4→int8). Оформить как новую миграцию (напр. `010_*`), а не переоткрывать 004.
  2. Код: `uk_management_bot/database/models/audit.py:16` → `Column(BigInteger, ...)`, чтобы ORM/`create_all` больше не воссоздавал колонку как int4.
  3. Проверить остальные int-колонки под Telegram ID на предмет той же проблемы (модель `audit.py` декларирует `Integer`, но `users.telegram_id` уже `bigint` — рассинхрон).
