# Синхронизация констант статусов заявок

**Дата**: 16 октября 2025
**Задача**: Синхронизировать константы статусов между файлами и обновить матрицу переходов
**Статус**: ✅ ЗАВЕРШЕНО

## Проблема

В системе наблюдалось несоответствие между различными файлами в определении статусов заявок:

1. **settings.py**: Дубликаты статусов, отсутствует "Исполнено"
2. **constants.py**: Использовался устаревший статус "Подтверждена" вместо "Принято"
3. **request_service.py**: Матрица переходов не содержала новые статусы "Исполнено" и "Принято"
4. **handlers**: Множественные упоминания устаревшего статуса "Подтверждена"

#### 3.1. Обновлен метод is_transition_allowed

```python
def is_transition_allowed(self, current_status: str, target_status: str) -> bool:
    """
    Матрица допустимых переходов статусов.

    Обновлено 16.10.2025:
    - Добавлены статусы "Исполнено" и "Принято"
    - Удален старый статус "Подтверждена"
    - Добавлен сценарий возврата заявки (Исполнено -> В работе)

    Workflow:
    Новая -> Принята -> В работе -> Выполнена -> Исполнено -> Принято
                            ↓         ↓           ↑
                       Уточнение ↔ Закуп         ↓
                                            (возврат заявителем)
    """
    allowed: Dict[str, List[str]] = {
        # Новая заявка: менеджер принимает и назначает или сразу переводит в работу
        "Новая": ["Принята", "В работе", "Закуп", "Уточнение", "Отменена"],

        # Принята: менеджер назначает исполнителю
        "Принята": ["В работе", "Отменена"],

        # В работе: исполнитель выполняет или запрашивает уточнение/закуп
        "В работе": ["Уточнение", "Закуп", "Выполнена", "Отменена"],

        # Уточнение: можно вернуть в работу или перейти к закупу
        "Уточнение": ["В работе", "Закуп", "Отменена"],

        # Закуп: после закупки возврат в работу или уточнение
        "Закуп": ["В работе", "Уточнение", "Отменена"],

        # ✅ Выполнена: исполнитель завершил, менеджер проверяет
        "Выполнена": ["Исполнено", "В работе", "Отменена"],

        # ✅ НОВОЕ: Исполнено - менеджер подтвердил и отправил заявителю
        # Заявитель может принять (Принято) или вернуть на доработку (В работе)
        "Исполнено": ["Принято", "В работе", "Отменена"],

        # ✅ НОВОЕ: Принято - финальный статус, заявитель принял работу
        "Принято": [],

        # Отменена: финальный статус
        "Отменена": [],
    }
    return target_status in allowed.get(current_status, [])
```

**Ключевые изменения**:
- ✅ Статус "Подтверждена" → "Принято"
- ✅ Добавлен статус "Исполнено" с переходами
- ✅ Добавлен переход "Исполнено" → "В работе" (возврат заявки)
- ✅ Добавлен переход "Исполнено" → "Принято" (принятие заявки)
- ✅ Добавлен переход "Выполнена" → "Исполнено" (подтверждение менеджером)
- ✅ Добавлена ASCII-диаграмма workflow в docstring

## Права на изменение статусов

### Заявитель (ROLE_APPLICANT)

| Из статуса | В статус | Условие | Описание |
|------------|----------|---------|----------|
| Новая | Отменена | is_owner | Отмена своей заявки |
| Исполнено | Принято | is_owner | Принятие выполненной работы |
| Исполнено | В работе | is_owner | Возврат на доработку (с флагом is_returned) |

### Исполнитель (ROLE_EXECUTOR)

| Доступные статусы для установки |
|--------------------------------|
| В работе |
| Уточнение |
| Закуп |
| Выполнена |

### Менеджер (ROLE_MANAGER) и Админ (admin)

Полные права на все переходы статусов согласно матрице

## Специальные механизмы

### Возврат заявки (is_returned = true)

Когда заявитель возвращает заявку на доработку:

1. **Статус**: "Исполнено" → "Исполнено" (не меняется при возврате через request_acceptance.py)
2. **Флаги**:
   - `is_returned = true`
   - `return_reason` = причина возврата
   - `return_media` = медиафайлы с пояснениями
   - `returned_by` = ID заявителя
   - `returned_at` = время возврата

3. **Фильтрация в UI**:
   - Возвращенные заявки фильтруются по `is_returned = true` AND `status = "Исполнено"`
   - См. [admin.py:1113](uk_management_bot/handlers/admin.py#L1113)

### Подтверждение менеджером

Когда менеджер подтверждает выполненную заявку:

1. **Статус**: "Выполнена" → "Исполнено"
2. **Флаги**:
   - `manager_confirmed = true`
   - `confirmed_by` = ID менеджера

## Миграция данных

### Существующие заявки со статусом "Подтверждена"

Если в базе данных есть заявки со статусом "Подтверждена", необходимо выполнить миграцию:

```sql
-- Обновить статус с "Подтверждена" на "Принято"
UPDATE requests
SET status = 'Принято'
WHERE status = 'Подтверждена';

-- Проверить результат
SELECT COUNT(*) FROM requests WHERE status = 'Подтверждена';
-- Должно вернуть 0
```
