## Обнаруженные и исправленные проблемы

### 1. ✅ Групповые назначения не создавались в БД

**Проблема**: При нажатии "Дежурный специалист" функция выполнялась, логи показывали успех, но запись в `request_assignments` не создавалась.

**Причина**: Функция `auto_assign_request_by_category()` находила 0 исполнителей с `active_role='executor'`, потому что у пользователя Andrey была активная роль `'manager'`.

**Решение**:
1. Исправлен `active_role` на `'executor'` для пользователя Andrey
2. Добавлено детальное логирование с префиксом `[AUTO_ASSIGN]` на каждом этапе
3. Расширен маппинг категорий (добавлены "Отопление", "Вентиляция")

**Файлы**: [admin.py:57-215](uk_management_bot/handlers/admin.py#L57)  
**Документация**: [FIX_GROUP_ASSIGNMENT_NOTIFICATIONS.md](FIX_GROUP_ASSIGNMENT_NOTIFICATIONS.md)

### 2. ✅ Список конкретных исполнителей был пуст

**Проблема**: При нажатии "Конкретный специалист" список показывал "Нет доступных исполнителей".

**Причины**:
1. Использовался `User.roles.contains('"executor"')` вместо `User.active_role == "executor"`
2. Неполный маппинг категорий (отсутствовали HVAC, Ремонт, Установка и др.)
3. Неправильное значение для "Уборка": `"cleaner"` вместо `"cleaning"`

**Решение**:
1. Заменён запрос на `User.active_role == "executor"`
2. Расширен маппинг категорий до 12 позиций
3. Синхронизирован с маппингом в `auto_assign_request_by_category()`
4. Добавлено логирование с префиксом `[SPECIFIC_ASSIGN]`

**Файлы**: [admin.py:2415-2463](uk_management_bot/handlers/admin.py#L2415)  
**Документация**: [FIX_ROLE_SELECTION_BUTTON.md](FIX_ROLE_SELECTION_BUTTON.md)

### 3. ✅ Foreign Key Constraint при назначении конкретного исполнителя

**Проблема**: 
```
ERROR: insert or update on table "requests" violates foreign key constraint "requests_assigned_by_fkey"
DETAIL: Key (assigned_by)=(48617336) is not present in table "users".
```

**Причина**: Использовался `callback.from_user.id` (telegram_id) вместо `user.id` (primary key) для поля `assigned_by`.

**Решение**:
1. Добавлен параметр `user: User = None` в функцию
2. Если `user` не передан - запрашивается из БД по `telegram_id`
3. Используется `user.id` вместо `callback.from_user.id`

**Файлы**: [admin.py:2487-2527](uk_management_bot/handlers/admin.py#L2487)  
**Документация**: [FIX_ASSIGNMENT_FOREIGN_KEY.md](FIX_ASSIGNMENT_FOREIGN_KEY.md)

### 4. ✅ MESSAGE_TOO_LONG при уведомлении исполнителя

**Проблема**: Уведомление исполнителю превышало лимит Telegram (4096 символов) из-за длинных адресов и описаний.

**Решение**:
1. Уменьшены лимиты: адрес до 150 символов, описание до 300 символов
2. Общий лимит уведомления снижен с 4000 до 3500 символов
3. Добавлено логирование длины сообщения перед отправкой

**Файлы**: [admin.py:2564-2589](uk_management_bot/handlers/admin.py#L2564)  
**Документация**: [FIX_ASSIGNMENT_FOREIGN_KEY.md](FIX_ASSIGNMENT_FOREIGN_KEY.md)

---

### 5. ✅ Роли в CSV-формате вместо JSON

**Проблема**: В профиле показывалось "нет ролей", потому что поле `roles` содержало CSV-строку `"applicant,executor,manager"` вместо JSON `["applicant", "executor", "manager"]`.

**Причина**: Тип столбца `roles` - `text`, а не `jsonb`. Код пытался распарсить CSV как JSON и получал ошибку.

**Решение**:
1. Конвертирован формат для пользователя Andrey: `'["applicant", "executor", "manager"]'`
2. Проверен код сохранения ролей - использует `json.dumps()` (корректно)
3. Рекомендовано не менять роли через UI до миграции типа столбца на `jsonb`

**Файлы**: Изменения в БД  
**Примечание**: Требуется миграция столбца `roles` с `text` на `jsonb` в будущем
