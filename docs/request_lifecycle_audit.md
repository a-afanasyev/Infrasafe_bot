**Дата аудита:** 2026-03-08

## 1. Статусы заявок

| Статус | Константа | Описание (задуманное) | Финальный? |
|--------|-----------|----------------------|------------|
| Новая | REQUEST_STATUS_NEW | Создана заявителем, ожидает назначения | Нет |
| В работе | REQUEST_STATUS_IN_PROGRESS | Назначена исполнителю, в процессе | Нет |
| Закуп | REQUEST_STATUS_PURCHASE | Требуется закупка материалов | Нет |
| Уточнение | REQUEST_STATUS_CLARIFICATION | Требуется уточнение деталей | Нет |
| Выполнена | REQUEST_STATUS_EXECUTED | Исполнитель завершил, ожидает менеджера | Нет |
| Исполнено | REQUEST_STATUS_COMPLETED | Менеджер подтвердил, отправлена заявителю | Нет |
| Принято | REQUEST_STATUS_APPROVED | Заявитель принял работу | **Да** |
| Отменена | REQUEST_STATUS_CANCELLED | Заявка отменена | **Да** |

## 3. Фактические бизнес-процессы (как реализовано)

### 3.1. Создание заявки

**Файл:** `handlers/requests.py`
**Роль:** Заявитель (applicant)
**Результат:** статус = "Новая"

**Пошаговый процесс (FSM: RequestStates):**

1. Заявитель нажимает "Создать заявку" → `RequestStates.category`
2. Выбор категории (8 вариантов) → `RequestStates.address`
3. Выбор адреса (двор/здание/квартира) → `RequestStates.description`
4. Ввод описания проблемы → `RequestStates.urgency`
5. Выбор срочности (4 уровня) → `RequestStates.media`
6. Прикрепление медиа (до 5 фото/видео, необязательно) → `RequestStates.confirm`
7. Подтверждение → создание записи в БД

**Данные заявки при создании:**
- request_number (YYMMDD-NNN, автогенерация)
- category (внутренний ключ: electricity, plumbing и т.д.)
- address, apartment_id/building_id/yard_id
- description, urgency
- media_files (JSON array)
- status = "Новая"
- user_id = ID заявителя

**После создания:**
- Автоназначение по категории (`auto_assign_request_by_category()`)
- Уведомление заявителю

---

### 3.2. Назначение исполнителя

**Файл:** `handlers/request_assignment.py`
**Роль:** Менеджер
**Переход:** Новая → В работе

**Два способа назначения:**

1. **Групповое** (`assign_group_*`):
   - Выбор специализации → подтверждение
   - Создается RequestAssignment с assignment_type="group"

2. **Индивидуальное** (`assign_individual_*`):
   - Выбор конкретного исполнителя из списка → подтверждение
   - Создается RequestAssignment с assignment_type="individual"
   - Устанавливается request.executor_id

**Уведомления:** исполнителю о назначении

---

### 3.3. Работа исполнителя

**Файл:** `handlers/request_status_management.py`
**Роль:** Исполнитель (executor)
**Условие:** Исполнитель должен быть в активной смене

**Доступные действия из статуса "В работе":**

| Действие | Переход статуса | Описание |
|----------|----------------|----------|
| Запрос закупки | В работе → Закуп | Ввод списка материалов |
| Запрос уточнения | В работе → Уточнение | Ввод вопроса заявителю |
| Завершение работы | В работе → ??? | **См. баг #1 ниже** |

**Из статуса "Закуп":**
- Возврат в работу → "В работе"

**Из статуса "Уточнение":**
- После ответа заявителя менеджер возвращает → "В работе"

**Завершение работы (completion report):**
1. Исполнитель выбирает "Завершить работу"
2. Прикрепляет медиа (до 5 файлов)
3. Вводит отчет о выполнении
4. Данные сохраняются в request.completion_report и request.completion_media

---

### 3.4. Подтверждение менеджером

**Файл:** `handlers/admin.py` (callback: `confirm_completed_*`)
**Роль:** Менеджер
**Действие:** Менеджер проверяет работу исполнителя

**Что происходит:**
- `request.status = "Выполнена"` (напрямую, без state machine)
- `request.manager_confirmed = True`
- `request.manager_confirmed_by = user.id`
- `request.manager_confirmed_at = datetime.now()`
- Уведомление заявителю: "Ваша заявка выполнена, примите ее"

---

### 3.5. Приемка заявителем

**Файл:** `handlers/request_acceptance.py`
**Роль:** Заявитель (applicant)

**Список ожидающих приемки:**
- Фильтр: `Request.status == "Выполнена"` AND `Request.user_id == user.id`
- **НЕ проверяет** `manager_confirmed`!

**Два варианта:**

#### A. Принятие (accept):
1. Заявитель выбирает заявку → видит отчет и медиа
2. Нажимает "Принять" → выбирает оценку (1-5 звезд)
3. Создается Rating(request_number, user_id, rating)
4. `request.status = "Принято"` (напрямую, **без state machine**)
5. `request.completed_at = datetime.now()`
6. Уведомления: заявителю, исполнителю, в канал

#### B. Возврат на доработку (return):
1. Заявитель нажимает "Вернуть"
2. Вводит причину возврата
3. Прикрепляет медиа (необязательно)
4. **Переход:** `request.status = "Исполнено"` (напрямую)
5. `request.is_returned = True`
6. `request.return_reason = причина`
7. `request.return_media = [file_ids]`
8. `request.manager_confirmed = False` (сброс)
9. Уведомления всем менеджерам о возврате

---

### 3.6. Непринятые заявки (менеджерский контроль)

**Файл:** `handlers/unaccepted_requests.py`
**Роль:** Менеджер

**Список непринятых:**
- Фильтр: `status == "Выполнена"` AND `manager_confirmed == True` AND `is_returned == False`

**Действия менеджера:**

| Действие | Описание |
|----------|----------|
| Напомнить заявителю | Отправка напоминания в Telegram |
| Принять от имени заявителя | Менеджер вводит обязательный комментарий (10+ символов), заявка переходит в "Принято" **без оценки** |

---

### 3.7. Повторное подтверждение (после возврата)

**Файл:** `handlers/admin.py` (callback: `reconfirm_completed_*`)
**Роль:** Менеджер

**Что происходит:**
- `request.status = "Выполнена"` (повторно)
- `request.is_returned = False` (сброс флага)
- `request.manager_confirmed = True`
- Уведомление заявителю: "Замечания учтены, примите заявку"

Далее цикл возвращается к п. 3.5 (приемка заявителем).

---

### 3.8. Ответ на уточнение

**Файл:** `handlers/clarification_replies.py`
**Роль:** Заявитель

1. Заявитель отправляет `/reply_<request_number>`
2. Статус должен быть "Уточнение"
3. Вводит ответ
4. Ответ добавляется в request.notes
5. **Статус НЕ меняется** (остается "Уточнение")
6. Уведомление всем менеджерам

---

### 3.9. Отмена заявки

**Переход:** Любой статус → "Отменена"
**Доступность по ролям:**
- **Заявитель:** может отменить только свою заявку в статусе "Новая"
- **Менеджер:** может отменить любую заявку из любого не-финального статуса

## 5. ВЫЯВЛЕННЫЕ НЕСООТВЕТСТВИЯ

### БАГ #1 (КРИТИЧЕСКИЙ): Путаница REQUEST_STATUS_COMPLETED / REQUEST_STATUS_EXECUTED

**Файл:** `handlers/request_status_management.py:549-553`

```python
# Изменяем статус на "Исполнено"
updated_request = request_service.update_status_by_actor(
    request_number=request_number,
    new_status=REQUEST_STATUS_COMPLETED,  # "Исполнено"
    actor_telegram_id=message.from_user.id
)
```

**Проблема:** Когда исполнитель завершает работу, обработчик пытается установить статус "Исполнено" (REQUEST_STATUS_COMPLETED). Но state machine разрешает из "В работе" только переход в "Выполнена" (REQUEST_STATUS_EXECUTED).

**Следствие:** `update_status_by_actor()` вызывает `is_transition_allowed("В работе", "Исполнено")` → возвращает `False`. Метод возвращает `{success: False}`. **Но обработчик не проверяет результат!** Он продолжает сохранять `completion_report` и вызывает `db.commit()`. В итоге:
- Отчет сохраняется в БД
- Статус НЕ меняется (остается "В работе")
- Пользователь получает сообщение "Работа завершена", хотя статус не изменился

**Такая же проблема в `get_available_statuses()`:**
- Строка 608: менеджеру предлагается "Исполнено" из "В работе"
- Строка 619: исполнителю предлагается "Исполнено" из "В работе"

Оба случая конфликтуют со state machine, которая разрешает только "Выполнена".

**Корневая причина:** Путаница в именовании констант:
- `REQUEST_STATUS_EXECUTED` = "Выполнена" (звучит как "выполнено" по-английски)
- `REQUEST_STATUS_COMPLETED` = "Исполнено" (звучит как "завершено" по-английски)

Разработчик, скорее всего, использовал COMPLETED ("завершено") по семантическому сходству, тогда как по workflow нужен EXECUTED.

**Рекомендация:** Заменить `REQUEST_STATUS_COMPLETED` на `REQUEST_STATUS_EXECUTED` в:
- `handlers/request_status_management.py:552` (завершение работы)
- `handlers/request_status_management.py:608` (доступные статусы для менеджера)
- `handlers/request_status_management.py:619` (доступные статусы для исполнителя)

---

### БАГ #2 (ВЫСОКИЙ): State machine обходится напрямую

Несколько обработчиков модифицируют `request.status` напрямую, минуя `update_status_by_actor()`:

| Файл | Строка | Переход | Проблема |
|------|--------|---------|----------|
| `handlers/admin.py` | 560 | → "Выполнена" | Менеджер подтверждает без проверки state machine |
| `handlers/admin.py` | 630 | → "Выполнена" | Повторное подтверждение без проверки |
| `handlers/request_acceptance.py` | 327 | "Выполнена" → "Принято" | **State machine НЕ разрешает** этот переход! (Выполнена → Принято отсутствует в матрице) |
| `handlers/request_acceptance.py` | 498 | → "Исполнено" | Возврат без проверки |
| `handlers/unaccepted_requests.py` | ~203 | → "Принято" | Менеджерская приемка без проверки |

**Следствие:** State machine в `is_transition_allowed()` фактически является "мертвым кодом" для половины переходов. Переходы не валидируются.

---

### БАГ #3 (ВЫСОКИЙ): Дублирование механизма подтверждения менеджером

В системе существуют **два параллельных механизма** подтверждения менеджером:

1. **Через статус:** "Выполнена" → "Исполнено" (state machine)
2. **Через флаг:** `manager_confirmed = True` (поле в модели Request)

Эти механизмы не синхронизированы:
- admin.py использует флаг (статус "Выполнена" + manager_confirmed)
- State machine ожидает отдельный статус "Исполнено"
- Приемка заявителем фильтрует по "Выполнена" (не по "Исполнено")

**Рекомендация:** Выбрать один механизм и привести все обработчики к единообразию.

### БАГ #4 (СРЕДНИЙ): Приемка без проверки manager_confirmed

**Файл:** `handlers/request_acceptance.py:63-68`


**Проблема:** Заявитель видит ВСЕ заявки со статусом "Выполнена", включая те, которые менеджер еще не подтвердил. Заявитель может принять заявку до проверки менеджером.

### БАГ #5 (СРЕДНИЙ): Результат update_status_by_actor не проверяется

**Файл:** `handlers/request_status_management.py:550-570`

```python
updated_request = request_service.update_status_by_actor(...)
# Нет проверки updated_request["success"]!
request.completion_report = full_report  # Выполняется в любом случае
db.commit()  # Коммит в любом случае
```

**Проблема:** Даже если смена статуса отклонена (success=False), отчет сохраняется, и пользователь получает подтверждение успеха.

### НЕСООТВЕТСТВИЕ #6 (НИЗКИЙ): Статус "Исполнено" используется двояко

1. **В state machine:** "Исполнено" = менеджер подтвердил выполнение (после "Выполнена")
2. **В возврате:** "Исполнено" = заявитель вернул заявку на доработку (`request_acceptance.py:498`)

Эти два использования конфликтуют:
- В первом случае "Исполнено" — шаг вперед
- Во втором — шаг назад (возврат)

---

### НЕСООТВЕТСТВИЕ #7 (НИЗКИЙ): Нет ограничений на цикл возвратов

Заявитель может бесконечно возвращать заявку:
"Выполнена" → возврат → "Исполнено" → менеджер re-confirm → "Выполнена" → возврат → ...

Нет лимита на количество возвратов и нет эскалации.

| status_change | При смене статуса через update_status_by_actor() |

| purchase | При запросе закупки |
| report | При завершении работы (отчет) |

### AuditLog:

| Действие | Данные |
|----------|--------|
| request_status_changed | request_number, old_status, new_status, notes, actor_role |

**Проблема:** Аудит создается только через `update_status_by_actor()`. Прямые изменения статуса в admin.py и request_acceptance.py **НЕ логируются** в AuditLog.
