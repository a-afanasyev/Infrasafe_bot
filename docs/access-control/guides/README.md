# Контроль доступа — документация и инструкции

Модуль **контроля въезда** (`access_control` / сервис `uk-access-api`) — отдельный
доменный сервис внутри монорепозитория УК: ANPR-камеры распознают номер, центральный
backend принимает решение, edge управляет шлагбаумом. Подробное ТЗ — в
[`../TECHNICAL_SPEC.md`](../TECHNICAL_SPEC.md) (v1.4).

## Инструкции по ролям

| Роль | Кому | Документ |
|---|---|---|
| 🏠 Житель (`applicant`) | жильцы ЖК | [Инструкция жителя](resident.md) — бот и Mini App: авто, пропуска, гостевые коды, спорный въезд |
| 👮 Охрана (`security_operator`) | операторы поста охраны | [Инструкция охраны](operator.md) — live-лента, ручная проверка, гостевые коды, ручное открытие |
| 👔 Менеджер / админ (`manager`, `system_admin`) | УК | [Инструкция менеджера](manager.md) — авто, заявки, пропуска, зоны/места, оборудование, история |
| 🛠 Технический обзор | разработка/эксплуатация | [Технический обзор](#технический-обзор) (ниже) |

## Технический обзор

### Состав
- **Сервис `uk-access-api`** (FastAPI) — REST + WebSocket, общая PostgreSQL/Redis с основным стеком.
- **Дашборд УК** (React) — разделы «Контроль доступа», «История проездов», «База доступа», «Оборудование».
- **Telegram-бот** (aiogram) — раздел жителя + уведомления.
- **TWA Mini App** — раздел «Доступ» для жителя.
- **Медиа-сервис** — хранение фото проезда (обзор + номер) в отдельном Telegram-канале.

### Ключевые принципы (ТЗ §4)
1. Камера только распознаёт номер; решение принимает backend.
2. Edge offline применяет только подписанный snapshot (пилот — `fail_closed`).
3. Клиенты (бот/TWA/web) не содержат бизнес-логики — единый API и Decision Engine.
4. События, решения, команды и аудит **идемпотентны** и **append-only** (нельзя изменить/удалить прикладной ролью).
5. Ручное открытие — только персональный аккаунт + причина.

### Роли (ТЗ §3.2)
`applicant` — житель · `security_operator` — охрана · `manager` — администратор УК ·
`system_admin` — системный администратор (оборудование, device-ключи).
`executor`/`inspector` доступа к модулю не имеют.

### Архитектура решения о въезде
```
ANPR-камера → edge → POST /camera-events/anpr (device-auth HMAC)
   → Decision Engine: нормализация номера → постоянный авто / taxi-pass / тип зоны
   → allow | deny | manual_review (+ причина)
   → при allow: команда в barrier_commands (outbox) → edge открывает шлагбаум
   → событие в live-ленте охраны (WebSocket), запись в append-only журнал
```

### Тип парковки (важно для логики)
- **Закреплённые места** (`assigned`): место куплено/арендовано за квартирой; пускаем
  любой активный авто квартиры с активным закреплением (аренда — по сроку).
- **Общие зоны** (`shared`): личных мест нет; пускаем все авто, зарегистрированные на
  квартиру, обслуживаемую зоной; гибкий лимит на квартиру; учёт заездов (полный контроль
  занятости — после оснащения выездных камер).

### Безопасность и ПД (ТЗ §9, §11)
- Device-auth: API-key + HMAC тела + nonce/timestamp + IP-allowlist (прод — mTLS).
- Фото/номера/коды — **не в обычных логах**; фото — по короткоживущим signed-URL с
  отдельным разрешением; одноразовые коды — только хэш, TTL ≤30 мин, rate-limit.
- Append-only журналы решений/событий/аудита защищены триггерами и DB-грантами.

### Состояние пилота
Синтетический стенд (реальные ПД и камера подключаются после выполнения §11 и
обследования объекта). Развёрнут на dev (`uk-access-api`), проверяется ANPR-симулятором и
функцией «Тест точки въезда».
