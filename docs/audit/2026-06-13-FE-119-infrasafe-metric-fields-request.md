# Запрос UK → InfraSafe: данные метрики/инфраструктуры в `alert.created`

**Дата:** 2026-06-13 · **Тикет UK:** FE-119 · **Тип:** запрос на расширение исходящего webhook-контракта · **Приоритет:** не блокер cutover'а

## Сообщение для отправки

Коллеги, привет!

По задаче на нашей стороне (FE-119) хотим показывать оператору УК в карточке заявки **причину** алерта — конкретное значение метрики и какой объект инфраструктуры затронут. Сейчас из вашего `alert.created` мы получаем `message`, `type`, `severity` (раскладываем в описание/категорию/срочность) и трассировку `alert_id` / `created_at` / `correlation_id`. Структурных данных метрики, на которой сработало правило, в payload нет.

Просим, если это технически доступно у вас в момент формирования алерта, добавить несколько optional-полей **внутрь блока `alert`** (так же, как уже сделаны `reopen_chain_id`, `engineer_required_reason`, `uk_urgency_override`).

Важно: можно слать **инкрементально, по мере готовности** — наша валидация игнорирует неизвестные ключи и читает поля из сырого payload, так что ничего не сломается, а UI оживёт автоматически, как только поле начнёт приходить.

### Что просим (всё optional, snake_case, внутри `alert`)

**Метрика (главное — ради этого тикет):**
| Ключ | Тип | Пример | Назначение |
|---|---|---|---|
| `metric_label` | string | `"Напряжение"` | человекочитаемое имя метрики для UI |
| `metric_value` | number | `215.3` | текущее значение, на котором сработало правило |
| `metric_unit` | string | `"В"` | единица измерения |
| `metric_normal_min` | number | `220` | нижняя граница нормы (если есть в правиле) |
| `metric_normal_max` | number | `240` | верхняя граница нормы (если есть в правиле) |
| `metric_id` | string | `"voltage_l1"` | стабильный ключ метрики (опц., если есть) |

Отрисуем как: **«Напряжение: 215.3 В (норма 220–240 В)»**

**Инфраструктура:**
| Ключ | Тип | Пример | Назначение |
|---|---|---|---|
| `infrastructure_type` | string | `"transformer"` | тип объекта |
| `infrastructure_id` | string | `"10"` | идентификатор объекта |
| `infrastructure_label` | string | `"Трансформатор №10"` | готовая подпись (опц.; иначе соберём из type+id) |

Отрисуем как: **«Трансформатор №10»**

### Вопросы к вам
1. Какие из этих полей **технически доступны** у вас в момент `alert.created`?
2. Подтвердите, пожалуйста, **точные имена ключей** — чтобы мы читали именно их, без догадок.
3. Граница нормы (`metric_normal_min/max`) — доступна ли из `alert_rule` на момент срабатывания?
4. Подтвердите вложенность **внутрь блока `alert`** (а не на top-level).

Как только подтвердите имена и состав — мы за один заход добавим чтение этих полей и блок в UI. Спасибо!
