### FIX-005 — Webhook HTTP 503 помечен `retryable=False`
- **Priority:** P0
- **Type:** BUG
- **Source:** P (C3)
- **Files:** `uk_management_bot/services/webhook_sender.py:189-190`
- **Description:** По коду `process_outbox` (lines 250-330) переход в `failed` срабатывает по условию `if not retryable or record.attempts >= max_retries`. То есть при `retryable=False` запись становится `failed` **сразу после первой неудачи**, не "после max_retries". Короткий 503 → перманентная потеря события.
- **Fix:** `return (False, "HTTP 503: service unavailable", True, 0)`.
- **AC:** Юнит-тест: при `httpx.Response(status_code=503)` запись после одного цикла `process_outbox` имеет `status="pending"`, `attempts=1` (поле модели — `attempts`, не `retry_count`), `retry_after` установлен; уходит в `failed` только после `attempts >= settings.INFRASAFE_WEBHOOK_MAX_RETRIES`.
- **Estimate:** 15 min.
- **Status:** done (2026-05-21, branch `fix/fix-005-webhook-503-retryable`, verifier log `docs/audit/verifier-logs/FIX-005.md`)

### FIX-007 — Inbound webhook router без HMAC verification — **gating-условие, не runtime-блокер**
- **Priority:** P0 (gating)
- **Type:** SEC / ARCH
- **Source:** S (C-3)
- **Files:** `uk_management_bot/config/settings.py:99-100` (secret уже выдан) + новый роутер
- **Description:** `UK_WEBHOOK_SECRET` объявлен в settings, но endpoint-приёмник от InfraSafe не реализован. Текущая поверхность атаки = 0 (обращение → 404). Пункт P0 как gate: реализация inbound должна **с первого коммита** включать HMAC-проверку, иначе появится временное окно без защиты. Также: значение `UK_WEBHOOK_SECRET` уже лежит на диске и должно быть ротировано вместе с остальными секретами (FIX-002).
- **Fix:** При реализации inbound endpoint — verify HMAC SHA-256 над сырым body перед обработкой; rotation через `UK_WEBHOOK_SECRET_NEXT`; replay protection через nonce/timestamp; rate-limit. Если до релиза inbound не делается — пункт допустимо понизить до P1 и снять с критического пути.
- **AC:** Endpoint без подписи → 401; с неверной подписью → 401; replay (тот же nonce 2 раза) → 409.
- **Estimate:** 4 h.

### FIX-008 — AddressService — 16 `async def` без `await` на sync Session — **tech-debt, не runtime-блокер**
- **Priority:** P0 (gating — см. ниже)
- **Type:** ARCH / REFACTOR
- **Source:** P (C4)
- **Files:** `uk_management_bot/services/address_service.py` (lines 23, 68, 75, 249, 263, 545, 561, 586, 780, 815, 850, 870, 893, 914, 944, 1036)

#### ARCH-010 — Outbox: недетерминированный `event_id`
- **Priority:** P1 — **Type:** ARCH — **Source:** A (H3)
- **Files:** `uk_management_bot/services/webhook_sender.py`
- **Description:** `event_id = uuid.uuid4()` per-call. Два разных вызова `queue_webhook` для одного entity (например, ручной retry оператора + reconciler через час) получают разные id → receiver не может сам отбросить дубликат без отдельной storage-логики (entity_id + updated_at сравнения).
- **Fix:** `event_id = sha256(f"{event}:{entity_id}:{entity_updated_at}")[:32]`. Это даёт идемпотентность по природе: один и тот же логический момент = один id. Receiver-side dedup (если он есть) продолжает работать; отсутствие receiver-side dedup перестаёт быть критичным.
- **AC:** Два вызова `queue_webhook(building.updated, building_id=X)` без изменения `updated_at` дают одинаковый `event_id`; после смены `updated_at` — разный.
- **Estimate:** 1 h.
