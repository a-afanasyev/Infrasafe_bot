# PR-7 (F-01) — verifier-log DBA-окна, 2026-07-15

Раскатка PostgreSQL least-privilege role separation на оба прода. План:
`~/.claude/plans/zesty-hugging-wozniak.md`. Ниже — что реально выполнено и
проверено, без секретов/паролей.

## Шаг 0 — read-only inventory (до окна)

Прогнан `scripts/dba_inventory.sql` на обоих продах. Итог: `profk_bot`/`uk_bot`
уже `NOSUPERUSER`/`NOCREATEROLE` (прежний FIX-004), но оставались владельцами
DB/schema/всех 110 relations — ровно то расхождение, которое должен был
устранить PR-7. Ни одна из новых ролей (`uk_migration_owner`, `uk_migrator`,
`uk_app_rw`, `uk_bot_runtime`, `uk_api_runtime`, `uk_access_runtime`) на момент
инвентаризации не существовала ни на одном проде — чистый старт, без частичного
дрейфа. `access_app_rw` уже реализовывала ровно ту 3-уровневую ACL-матрицу
(immut/other/shared), которую предполагает `acl_reconcile.py`. Единственное
расхождение между продами — recovery-суперюзер `uk_admin`: на profk
`CREATEDB=f, CREATEROLE=f`, на infrasafe/105 оба `t` (не блокирует, взято на
заметку).

## Находка и фикс во время окна: `ALTER SEQUENCE ... OWNER TO` для linked-sequence

Первый прогон `dba_ownership_transfer.sql` на profk упал:

```
ERROR:  cannot change owner of sequence "users_id_seq"
DETAIL:  Sequence "users_id_seq" is linked to table "users".
```

PostgreSQL запрещает менять владельца sequence, связанной с таблицей через
serial/identity-колонку, напрямую — владение синхронизируется автоматически
как побочный эффект `ALTER TABLE ... OWNER TO`. Проверено: во всей схеме 53 из
53 sequences линкованы (0 standalone). Транзакция была в `BEGIN...` (без
`COMMIT`) — упавший скрипт откатился автоматически при закрытии psql-сессии,
проверено запросом `pg_get_userbyid` сразу после (owner всё ещё `profk_bot`,
без изменений). Фикс — коммит `5d655ce`: убран `relkind='S'` из цикла явного
`ALTER ... OWNER TO`, sequences больше не перебираются отдельно. Повторный
прогон на profk и первый же прогон на infrasafe/105 (уже с фиксом) прошли
чисто.

## Провижининг — обходной путь через `uk_admin`

`provision_roles.sh` в задуманном виде подключается как `$POSTGRES_USER`
(`profk_bot`/`uk_bot`) — но у обеих этих ролей уже нет `CREATEROLE` (см. Шаг
0), поэтому `CREATE ROLE uk_migration_owner` падал `permission denied to
create role`. Обходной путь (на обоих продах): т.к. официальный образ
`postgres:15-alpine` по умолчанию доверяет (`trust`) локальным
unix-socket-подключениям, `docker exec uk-postgres psql -U uk_admin -d <db>`
работает без пароля — воспроизведена логика `provision_roles.sh`
(генерация 4 паролей через `/dev/urandom`+`od`, staging-запись
`.env.<role>.staging`, `provision_roles.sql` через stdin, атомарный
`mv` staging→live) вручную поверх этого подключения вместо контейнера
`provision-roles` (который жёстко использует TCP+`PGPASSWORD` от
`$POSTGRES_USER`). Follow-up: `provision_roles.sh` в репозитории по-прежнему
рассчитан на суперюзер-`$POSTGRES_USER`; на любом окружении, где
`POSTGRES_USER` уже демоутнут (как здесь), потребуется тот же обходной путь —
стоит рассмотреть отдельным PR явный `ADMIN_DATABASE_URL`-параметр вместо
implicit `$POSTGRES_USER`.

## Второе расхождение хостов: `docker compose build` и `env_file`

На profk (`docker compose` v5.3.0) `build api access-api app` отработал до
`provision-roles`, не требуя существования `.secrets/roles/.env.*`. На
infrasafe/105 (`docker compose` v2.32.4) та же команда упала: `env file
.../.env.api not found` — версия validates `env_file:` существование даже для
`build`. Обойдено переменой порядка: `provision-roles` → `build` на
infrasafe/105 (провижининг не зависит от собранных образов).

## Последовательность, выполненная на каждом проде (после фиксов выше)

1. `git push`/`git pull` до коммита `5d655ce`.
2. `.env.postgres` заведён из существующих `POSTGRES_DB/USER/PASSWORD` (без
   смены значений — просто перенос, откуда читается).
3. Backup: `pg_dump -Fc` (полный) + `pg_dumpall --globals-only`, сохранены в
   `~/uk-pg-backups/pr7-20260715/` на каждом хосте.
4. `docker compose build api access-api app migrate`.
5. `provision-roles` (через `uk_admin`, см. выше) → 4 роли/пароля,
   `.secrets/roles/.env.{migrate,bot,api,access}`.
6. Пост-чек: подключение под каждым из 4 credentials, `SELECT session_user`
   совпал с ожидаемой ролью — на обоих продах, все 4/4 OK.
7. Quiesce: `docker compose stop app api access-api` — начало даунтайма.
8. `dba_ownership_transfer.sql` через `uk_admin` — owner схемы/всех
   объектов → `uk_migration_owner`; bulk-грант `uk_app_rw` на не-access-domain
   таблицы; `REVOKE ALL ON SCHEMA public FROM PUBLIC` + `GRANT USAGE` для
   `uk_app_rw`/`access_app_rw`. Проверено: `access_app_rw` на `access_events`
   осталась `INSERT,SELECT` (не задета), `uk_app_rw` на `access_events` —
   пусто (правильно исключена), `uk_app_rw` на `users` — полный DML.
9. `docker compose run --rm --name uk-migrate migrate` — `alembic upgrade
   head` (no-op, уже на `003`) → `acl_reconcile.py` (carve-out
   `alembic_version` + revoke access-domain у `uk_app_rw`) → `alembic check`
   («No new upgrade operations detected» на обоих).
10. `docker compose up -d app api access-api` — фактический cutover.
11. Smoke: все три контейнера `healthy`, `db_preflight: schema up to date
    (alembic head='003')` в логах api/access-api, `/uk/health` 200 на обоих,
    бот отправил стартовое уведомление админу.
12. Негативные ACL-тесты (оба прода, идентичный результат):
    - `bot_runtime`: `CREATE TABLE` — denied; `SET ROLE uk_migration_owner` —
      denied; `SELECT` на `access_events` — denied; `UPDATE alembic_version` —
      denied.
    - `access_runtime`: `SELECT` на `access_events` — OK; `UPDATE` на
      `access_events` — denied (append-only); `CREATE ROLE` — denied.
    - `pg_stat_activity` после cutover: только `uk_admin`/`uk_bot_runtime`/
      `uk_api_runtime` (и `uk_access_runtime` при реальном обращении) —
      старая объединённая роль (`profk_bot`/`uk_bot`) в активных подключениях
      не участвует.
