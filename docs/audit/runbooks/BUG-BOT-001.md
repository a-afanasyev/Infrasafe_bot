# Runbook — BUG-BOT-001: BOT_USERNAME mismatch

## Problem

`BOT_USERNAME` previously defaulted to `"infrasafebot"` in
`uk_management_bot/config/settings.py`. The real bot username, derived from
`BOT_TOKEN` via Telegram `getMe()`, was `Work_space_away_bot`.

When `BOT_USERNAME` did not match the real username, every invite link the bot
generated (`https://t.me/<BOT_USERNAME>?start=...`) pointed users to a **wrong
production bot**. The discrepancy was silent — no log line, no error.

## Fix (already in code)

1. `config/settings.py` — `BOT_USERNAME = os.getenv("BOT_USERNAME")` (no
   default; may be `None`).
2. `main.py` — after `Bot(...)` init, the startup calls `await bot.get_me()`
   and:
   - if `BOT_USERNAME` is unset → logs `ERROR` and populates
     `settings.BOT_USERNAME` from the token-derived username (graceful);
   - if `BOT_USERNAME` differs from `getMe().username` → logs `ERROR` with
     both values;
   - if they match → logs `BOT_USERNAME OK: <username>`.

The bot never crashes from this check — diagnostic only.

## Operator steps (user action required)

### 1. Confirm the real bot username

Run inside the running container:

```bash
docker exec uk-management-bot python3 -c "
import asyncio
from aiogram import Bot
from uk_management_bot.config.settings import settings

async def main():
    bot = Bot(token=settings.BOT_TOKEN)
    me = await bot.get_me()
    print('username:', me.username)
    await bot.session.close()

asyncio.run(main())
"
```

Record the printed `username`. Do **not** print or share `BOT_TOKEN`.

### 2. Update `.env` and the production template

In `.env` (local/dev) add or update:

```
BOT_USERNAME=<value from step 1>
```

In `.env.production.template` add the same key with a placeholder so deploys
do not forget it:

```
BOT_USERNAME=<your_bot_username>
```

### 3. Restart the bot and verify

```bash
docker compose restart uk-management-bot
docker logs uk-management-bot --tail 50 | grep -E 'BOT_USERNAME|BUG-BOT-001'
```

Success looks like:

```
INFO ... BOT_USERNAME OK: <your_bot_username>
```

Failure to fix (still mismatched or unset) looks like:

```
ERROR ... BUG-BOT-001: BOT_USERNAME mismatch — configured='X', actual (getMe)='Y'.
```

or

```
ERROR ... BUG-BOT-001: BOT_USERNAME is not set in environment. Falling back ...
```

If you see either ERROR line, repeat step 2 with the correct value.

## Verification (acceptance)

- Startup log contains `BOT_USERNAME OK: <real username>`.
- An invite link generated by the bot opens the **correct** bot in Telegram.
- Unit test `uk_management_bot/tests/test_settings.py::test_bot_username_*`
  passes.

## Rollback

No rollback required — the change is purely additive (no default,
diagnostic log). If the check itself misbehaves, revert the `await
bot.get_me()` block in `main.py` to restore prior behaviour.
