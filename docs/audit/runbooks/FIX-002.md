## Step 1 — Revoke токенов в @BotFather (ручное)

1. Открыть `@BotFather` в Telegram.
2. `/mybots` → выбрать **первого бота** (`BOT_TOKEN` из root `.env`) → `API Token` → **Revoke current token** → скопировать НОВЫЙ токен.
3. То же для **второго бота** (`BOT_TOKEN` из `uk_management_bot/.env` — если это другой bot_id, проверь до revoke:
   ```bash
   curl -sS "https://api.telegram.org/bot<ROOT_TOKEN>/getMe" | jq '.result.id'
   curl -sS "https://api.telegram.org/bot<UK_TOKEN>/getMe" | jq '.result.id'
   ```
   Если `result.id` совпадает — у тебя один токен в двух файлах, revoke 1 раз и подставь в обе строки.).
4. То же для **media-бота** (`MEDIA_BOT_TOKEN` и/или `TELEGRAM_BOT_TOKEN`).

**Сохрани новые значения временно в локальный текстовый файл вне репо** (например, `~/.uk-secret-backups/$TS/new-tokens.txt`). НЕ публиковать, НЕ скриншотить, НЕ копировать в чаты.

## Step 2 — Сгенерировать новые криптосекреты

```bash
# Сгенерировать 4 значения, перенаправить в защищённый временный файл
TS=$(date -u +"%Y%m%dT%H%M%SZ")
TMP=~/.uk-secret-backups/$TS/new-cryptosecrets.txt
touch "$TMP" && chmod 600 "$TMP"

{
  echo "INVITE_SECRET=$(openssl rand -base64 32)"
  echo "INFRASAFE_WEBHOOK_SECRET=$(openssl rand -base64 32)"
  echo "UK_WEBHOOK_SECRET=$(openssl rand -base64 32)"
  echo "JWT_SECRET=$(openssl rand -base64 32)"
} > "$TMP"

# Сгенерировать ADMIN_PASSWORD ≥16 raw chars (не base64, чтобы не зависеть от SEC-083 URL-decode length gap)
echo "ADMIN_PASSWORD=$(openssl rand -hex 12)" >> "$TMP"   # 24 hex chars = ≥16

ls -la "$TMP"
echo "Новые значения в $TMP — НЕ публиковать."
```

`-hex 12` даёт 24 raw chars — точно ≥16 в decoded form. Альтернатива: `tr -dc 'A-Za-z0-9!@#$%^&*' </dev/urandom | head -c 20` для разнообразия символов — **если используешь альтернативу с спецсимволами, оборачивай значение в одинарные кавычки в `.env` (`ADMIN_PASSWORD='...'`)**, чтобы shell-парсер dotenv не интерпретировал `$`, `!`, `*`.

**Caveat по base64-секретам (`INVITE_SECRET`, `*_WEBHOOK_SECRET`, `JWT_SECRET`):** `openssl rand -base64 32` выдаёт символы `+ / =`. Эти секреты используются как HMAC-ключи / Pydantic-settings, не как URL-параметры → проблемы SEC-083 их не касаются. Но если какой-то из них в будущем потребуется передавать в URL, генерируй через `-hex 32` вместо `-base64 32`.

## Expected user-visible incidents (read BEFORE rotation)

Hot-swap каждого секрета имеет конкретные последствия, которые нельзя избежать без архитектурных изменений (ARCH-107 для JWT, ARCH-108 для ownership). Знай заранее:

| Секрет | Что произойдёт в момент rotation |
|---|---|
| `JWT_SECRET` | **Все активные web/TWA сессии force-logout одновременно**. Сейчас нет `kid`-header / dual-key verifier → graceful rotation невозможна. Уведоми active users заранее или планируй на low-traffic окно. Tracked: **ARCH-107 (P2)** для graceful JWT rotation в будущем. |
| `INVITE_SECRET` | **Все уже разосланные invite-ссылки моментально невалидны** (TTL часы-дни). Перед rotation полезно: `docker exec uk-postgres psql -U uk_bot -d uk_management -c "SELECT created_at, role FROM invite_nonces WHERE expires_at > now();"` — увидишь сколько invites придётся переоформить. |
| `INFRASAFE_WEBHOOK_SECRET` | См. **Step 2.5** ниже — есть proper dual-secret flow в коде, не делай hot-swap здесь. |
| `ADMIN_PASSWORD` | Активные admin-сессии получат 401 на следующем запросе (FastAPI читает settings.ADMIN_PASSWORD при каждой auth-проверке). Logout + re-login обязателен. |
| `BOT_TOKEN` × 2, `MEDIA_BOT_TOKEN` | Revoke в @BotFather **необратим**. Rollback `.env` не вернёт старый токен в работающее состояние. |
| `UK_WEBHOOK_SECRET` | Сейчас inbound endpoint не реализован (FIX-007 отложен) → blast radius нулевой. После реализации — потребуется dual-secret flow аналогичный INFRASAFE. |

**Suggested rotation window:** ~30 sec downtime бота + API на Step 5 (`docker compose down` пересоздаёт контейнеры; in-flight outbox events drain'ятся). Плюс ~5-10 min на админ/operator reconnect и переоформление активных invites.

## Step 2.5 — InfraSafe dual-secret rotation (proper flow)

**ВАЖНО:** Для `INFRASAFE_WEBHOOK_SECRET` НЕ используй hot-swap из Step 3. Код уже реализует textbook rotation через `INFRASAFE_WEBHOOK_SECRET_NEXT` + `INFRASAFE_USE_NEXT_SECRET` flag (см. `uk_management_bot/services/webhook_sender.py:200-209`). Hot-swap сломает HMAC у in-flight outbox events.

### Шаги:

1. **Скоординируйся с InfraSafe operator** (вы должны принимать OLD‖NEW HMAC ~15 мин в их verifier'е).

2. **Поставь NEW секрет рядом с OLD** в root `.env`, ничего НЕ удаляя:
   ```bash
   # В .env добавь две строки (НЕ заменяй существующий INFRASAFE_WEBHOOK_SECRET):
   echo "INFRASAFE_WEBHOOK_SECRET_NEXT=$(grep ^INFRASAFE_WEBHOOK_SECRET= ~/.uk-secret-backups/$TS/new-cryptosecrets.txt | cut -d= -f2-)" >> .env
   echo "INFRASAFE_USE_NEXT_SECRET=false" >> .env
   ```

3. **Restart бота**, проверь что OLD продолжает работать:
   ```bash
   docker compose restart uk-management-bot
   docker logs uk-management-bot --tail 30
   # Триггерни выдачу события (e.g., создание здания) → InfraSafe должен получить с OLD signature
   ```

4. **Флипни на NEW:**
   ```bash
   sed -i.bak 's/^INFRASAFE_USE_NEXT_SECRET=false$/INFRASAFE_USE_NEXT_SECRET=true/' .env
   docker compose restart uk-management-bot
   # Новые исходящие webhooks подписываются NEW. Триггерни ещё событие, верифицируй у InfraSafe.
   ```

5. **Drain outbox** — подожди пока все pending события переподпишутся под NEW (или вручную retry'нь failed):
   ```bash
   docker exec uk-postgres psql -U uk_bot -d uk_management -c "SELECT count(*) FROM webhook_outbox WHERE status='pending';"
   # Когда 0 — drain завершён.
   ```

6. **Promote NEW → primary:** замени `INFRASAFE_WEBHOOK_SECRET` на значение NEW, удали `_NEXT` и `USE_NEXT_SECRET`:
   ```bash
   ${EDITOR:-vim} .env   # переписать INFRASAFE_WEBHOOK_SECRET=<new>; удалить _NEXT и USE_NEXT
   docker compose restart uk-management-bot
   ```

7. **Скоординируйся с InfraSafe** на закрытие grace window (они снимают OLD из verifier'а).

Только после этого Step 3 ниже обрабатывает остальные 7 секретов (для них dual-secret машинерии нет → hot-swap единственный вариант, см. blast radius выше).

## Step 5 — Restart сервисов

**Ожидаемый downtime:** ~20-30 sec бота + API. `docker compose down` пересоздаёт контейнеры (не просто рестарт), in-flight `process_outbox` цикл прервётся → события доберут на retry после поднятия (это OK после FIX-005). В этот промежуток Telegram-вебхуки могут получить 502/connection-refused; aiogram-side retries закрывают это окно. Schedule на low-traffic window если возможно.

**Альтернатива (минимизировать downtime):** `docker compose restart` вместо `down` + `up` — переподнимает контейнеры в-месте, общий downtime ~5-10 sec. Используй если pull новых образов не нужен.

```bash
docker compose down
docker compose up -d uk-postgres uk-redis
sleep 5
docker compose up -d uk-management-bot uk-management-api uk-frontend
sleep 10
docker compose ps
docker logs uk-management-bot --tail 30
docker logs uk-management-api --tail 30
docker logs uk-media-service --tail 30 2>/dev/null || echo "(media-service container name may differ — check 'docker compose ps')"
```

**Smoke-check:**
- `docker exec uk-management-bot python -c "from uk_management_bot.config.settings import settings; print('BOT loaded:', bool(settings.BOT_TOKEN))"` → должно вывести `True`
- Отправить `/start` боту в Telegram → должен ответить
- Открыть admin-панель → войти с новым `ADMIN_PASSWORD`

## Rollback план

Если что-то пошло не так в течение 24 часов:

```bash
TS=<выбранный таймстемп бэкапа>
cp ~/.uk-secret-backups/$TS/root.env .env
cp ~/.uk-secret-backups/$TS/uk_management_bot.env uk_management_bot/.env
cp ~/.uk-secret-backups/$TS/media_service.env media_service/.env
docker compose restart uk-management-bot uk-management-api
```

**Caveat — что НЕ откатывается через rollback `.env`** (необратимые операции):

| Секрет | Почему необратим |
|---|---|
| `BOT_TOKEN` × 2, `MEDIA_BOT_TOKEN` | Revoked в @BotFather — старое значение мертво. Нужно сгенерировать ЕЩЁ один новый токен и подставить (rollback бесполезен). |
| `JWT_SECRET` | Все сессии, выпущенные на OLD ключе, уже инвалидированы клиентами (cookies дропнуты, TWA tokens force-logout произошёл). Rollback к OLD не воскресит сессии. |
| `INVITE_SECRET` | Все invite-ссылки, которые resident'ы успели кликнуть в окне rotation, уже либо отработали (consumed nonce), либо упали с "Invalid token signature". Rollback бесполезен — придётся переоформить invites. |
| `INFRASAFE_WEBHOOK_SECRET` | Если делал через Step 2.5 — rollback простой (вернуть `USE_NEXT_SECRET=false`). Если hot-swap — у InfraSafe verifier уже на NEW, нужна повторная координация. |
| `UK_WEBHOOK_SECRET`, `ADMIN_PASSWORD` | Обратимы через rollback `.env` (нет третьих сторон, нет invalidated state). |

**Используй rollback только если что-то ломается в первые ~5 минут** (permission denied, бот не стартует, миграции не идут). Через час+ rollback на обратимые секреты теряет смысл — лучше fix-forward.
