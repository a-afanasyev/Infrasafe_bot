**Implication:** уверенно демотим, ownership и schema-level privs сохраняются — alembic upgrade должен продолжать работать. Главный риск — отсутствие второго superuser'а: ставим **временный admin до демотa**.

### Важный caveat — что FIX-004 закрывает И что НЕ закрывает

FIX-004 **снижает**, но **не устраняет** destructive-DDL risk:

| После демота uk_bot МОЖЕТ | После демота uk_bot НЕ МОЖЕТ |
|---|---|
| `DROP TABLE`, `TRUNCATE`, `ALTER TABLE DROP COLUMN` любого объекта в `uk_management` (т.к. owner) | `DROP DATABASE uk_management` |
| `INSERT`/`UPDATE`/`DELETE`/`SELECT` любых строк | `CREATE ROLE`, `ALTER ROLE`, `CREATE USER` |
| Создавать новые таблицы/индексы в `public` (owner privileges) | `CREATE DATABASE`, `DROP DATABASE` other |
| Читать `pg_catalog` (как любой role) | Bypass any RLS / read superuser-only views |

**SQLi атака с возможностью `DROP TABLE requests` остаётся реалистичной** — owner-level права позволяют. Полное закрытие требует **reassign ownership на `uk_admin` + grant `uk_bot` только DML (SELECT/INSERT/UPDATE/DELETE)** — это отдельный architectural item, tracked как **ARCH-108 (P2)** в backlog.

FIX-004 — необходимый, но не достаточный шаг. Запускай его сейчас (он закрывает 70% blast radius), но не считай за full remediation.

## Step 1 — Bootstrap нового admin'а (`uk_admin`)

**Зачем:** после демотa `uk_bot` уже не сможет выпускать `ALTER ROLE`. Нужен fallback-superuser на случай recovery.

```bash
# Сгенерировать сильный пароль для uk_admin и положить в защищённый файл
PG_ADMIN_PWD=$(openssl rand -hex 16)
echo "PG_ADMIN_PWD=$PG_ADMIN_PWD" > ~/.uk-pg-backups/$TS/uk-admin-pwd.txt
chmod 600 ~/.uk-pg-backups/$TS/uk-admin-pwd.txt
echo "Пароль uk_admin сохранён в ~/.uk-pg-backups/$TS/uk-admin-pwd.txt — переложи в менеджер паролей."
echo "После переноса в vault — обязательно: shred -uvz ~/.uk-pg-backups/$TS/uk-admin-pwd.txt"

# Создать роль (выполняем под текущим superuser'ом uk_bot, пока ещё можно)
docker exec uk-postgres psql -U uk_bot -d postgres -c \
  "CREATE ROLE uk_admin WITH SUPERUSER CREATEDB CREATEROLE LOGIN PASSWORD '$PG_ADMIN_PWD';"

# Verify
docker exec uk-postgres psql -U uk_bot -d uk_management -c \
  "SELECT rolname, rolsuper, rolcreatedb, rolcreaterole FROM pg_roles WHERE rolname='uk_admin';"
# Ожидается: uk_admin | t | t | t
```

**В проде** (если применяешь к production-БД):
- Подключение через bastion / cloud SQL CLI / pgcli — выбери метод, под которым обычно делаешь admin-операции.
- `uk_admin` пароль положить в production-secret-store (Vault/AWS-SM/1Password), **не** в `.env` проекта.

## Step 3 — Smoke: alembic upgrade head после демотa

```bash
# alembic подключается как uk_bot (через DATABASE_URL). Прогон должен пройти,
# т.к. uk_bot — owner DB и schema, у него есть все table/sequence/schema privs.
docker exec uk-management-bot bash -c "cd /app && alembic current"
docker exec uk-management-bot bash -c "cd /app && alembic upgrade head"
# Должно вывести: "Running upgrade ..." и в итоге показать текущий head без ошибок.
```

Если упрётся в "permission denied for sequence" или "permission denied for schema":
```bash
# Восполняем GRANT'ы — единократно:
docker exec uk-postgres psql -U uk_admin -d uk_management -c \
  "GRANT USAGE, CREATE ON SCHEMA public TO uk_bot;
   GRANT SELECT, INSERT, UPDATE, DELETE, REFERENCES, TRIGGER ON ALL TABLES IN SCHEMA public TO uk_bot;
   GRANT USAGE, SELECT, UPDATE ON ALL SEQUENCES IN SCHEMA public TO uk_bot;
   GRANT EXECUTE ON ALL FUNCTIONS IN SCHEMA public TO uk_bot;
   ALTER DEFAULT PRIVILEGES IN SCHEMA public GRANT SELECT, INSERT, UPDATE, DELETE ON TABLES TO uk_bot;
   ALTER DEFAULT PRIVILEGES IN SCHEMA public GRANT USAGE, SELECT, UPDATE ON SEQUENCES TO uk_bot;
   ALTER DEFAULT PRIVILEGES FOR ROLE uk_admin IN SCHEMA public
     GRANT SELECT, INSERT, UPDATE, DELETE ON TABLES TO uk_bot;
   ALTER DEFAULT PRIVILEGES FOR ROLE uk_admin IN SCHEMA public
     GRANT USAGE, SELECT, UPDATE ON SEQUENCES TO uk_bot;"
```

**Что добавляет `FOR ROLE uk_admin`-вариант:** если в будущем кто-то выполнит DDL под `uk_admin` (emergency fix, ручной hotfix), новосозданные таблицы будут owned by `uk_admin`. Без `FOR ROLE` блока выше `uk_bot` к ним доступа не получит. Best practice: **все плановые DDL запускать под `uk_bot` post-демотa, `uk_admin` только emergency-only** — тогда ownership остаётся консистентной у `uk_bot`.

Затем повторить `alembic upgrade head`.

## Step 6 — Hardening (через 24-48 ч стабильной работы)

После того как demote выдержал боевую нагрузку (включая хотя бы одну успешную alembic-миграцию через прод-pipeline), сделай `uk_admin` "холодной запасной ролью".

### ⚠ Pre-condition: проверь, что есть external recovery path

**Без этой проверки можно лочнуть себя из кластера навсегда.** После Step 6 ни `uk_bot` (NOSUPERUSER), ни `uk_admin` (NOLOGIN) не смогут изменить роли. Единственный recovery — через `POSTGRES_USER`, заданный в `docker-compose.yml`.

```bash
# 1. Проверь docker-compose.yml на POSTGRES_USER
grep -E "^\s*-?\s*POSTGRES_USER" docker-compose.yml
# Если значение НЕ uk_bot (например 'postgres' или другая роль) — recovery возможен через
#   docker exec -it uk-postgres psql -U <это-значение>
# Если значение === uk_bot — НЕ ЗАПУСКАЙ STEP 6. Сначала создай второй emergency
# superuser (pg_break_glass) с надёжным паролем в vault.

# 2. Если есть альтернативный admin — verify recovery работает СЕЙЧАС, до hardening:
docker exec uk-postgres psql -U <POSTGRES_USER_value> -d postgres -c "SELECT current_user, rolsuper FROM pg_roles WHERE rolname=current_user;"
# Должно показать current_user + rolsuper=t.
```

Если pre-condition не прошёл — пропусти Step 6, оставь `uk_admin` в LOGIN-режиме и:
- **Никогда** не используй `uk_admin` для рутинных запросов (только для emergency)
- Добавь алерт в monitoring на `pg_stat_activity` с `usename='uk_admin'` (любое подключение под admin = инцидент)

### Step 6.A — Hardening (только после Pre-condition)

```bash
docker exec uk-postgres psql -U uk_admin -d postgres -c "ALTER ROLE uk_admin NOLOGIN;"
# Verify
docker exec uk-postgres psql -U uk_bot -d uk_management -c \
  "SELECT rolname, rolsuper, rolcanlogin FROM pg_roles WHERE rolname='uk_admin';"
# Ожидается: uk_admin | t | f  (super=on, но login=off)
```

### Recovery (когда нужно вернуть uk_admin в работу для admin-операции)

**ВАЖНО — правильный путь:** `uk_bot` (демотированный) уже **НЕ может** вернуть `uk_admin` в `LOGIN` — у него нет `CREATEROLE`/`SUPERUSER`, команда упадёт с *"must be superuser to alter superuser roles"*.

```bash
# Используй POSTGRES_USER (или pg_break_glass, если создал такого):
docker exec uk-postgres psql -U <POSTGRES_USER_value> -d postgres -c "ALTER ROLE uk_admin LOGIN;"
# выполняешь нужное под uk_admin, потом снова NOLOGIN:
docker exec uk-postgres psql -U uk_admin -d postgres -c "ALTER ROLE uk_admin NOLOGIN;"
```

**Если ни POSTGRES_USER, ни pg_break_glass недоступны** (catastrophic lockout):
- Остановить контейнер postgres
- Стартануть с `POSTGRES_HOST_AUTH_METHOD=trust` (без auth) — _только_ из консоли хоста, не из приложения
- Подключиться, выполнить `ALTER ROLE uk_admin LOGIN`
- Перезапустить с нормальным auth

Это болезненный путь — поэтому Pre-condition выше обязателен.

## Rollback план

Если в течение 24 ч что-то падает по permission:

```bash
docker exec uk-postgres psql -U uk_admin -d postgres -c \
  "ALTER ROLE uk_bot SUPERUSER CREATEDB CREATEROLE;"

docker exec uk-postgres psql -U uk_admin -d uk_management -c \
  "SELECT rolname, rolsuper FROM pg_roles WHERE rolname='uk_bot';"
# Должно показать rolsuper=t.
```

После rollback'а — открой issue с описанием ошибки и какой alembic step / endpoint падал, чтобы понять чего не хватило в GRANT'ах. Не оставляй уязвимое состояние "ещё не до конца раскопались" длительно — это и есть та самая SEC risk, которую FIX-004 закрывает.

## Production-specific notes

- На prod выполняй те же шаги, **но**:
  - `uk_admin` пароль кладём в production-secret-store, не в `~/.uk-pg-backups/`
  - Заранее проверь, нет ли уже отдельного admin-role (e.g., `rds_superuser` в AWS RDS — там твоя app-роль и так может быть NOSUPERUSER), `\du` подскажет
  - Backup перед операцией — full DB dump через `pg_dump`, не только `--schema-only`
  - Запланируй на low-traffic window (ALTER ROLE не блокирует, но если что-то отвалится, лучше иметь время на rollback)
