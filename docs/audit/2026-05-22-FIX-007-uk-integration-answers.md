## A. Схема `alert` (Phase 2)

🔶 Фаза 1 принимает `alert` как **произвольный объект** (`alert: dict`) — не
валидирует поля. Ниже — **предложение UK** по обязательному набору для Фазы 2;
финализируется при дизайне Фазы 2.

**A1. Поля `alert` для Phase 2:**

| Поле | UK-вердикт | Зачем |
|---|---|---|
| `external_id` (UUID здания UK) | ✅ **обязательно** | UK резолвит здание по нему. UK индексирует здания по детерминированному external_id = `SHA-256("uk-building-{id}")` (см. `reconciliation.py:_expected_external_id`). Шлите его. |
| `type` (`voltage_low`, ...) | ✅ **обязательно** | UK мапит `type` → `category` |
| `severity` | ✅ **обязательно** | UK мапит `severity` → `urgency` |
| `message` (RU) | ✅ **обязательно** | становится `description` заявки |
| `alert_id` (InfraSafe int) | 🔶 желательно | трассировка/debug, UK сохранит |
| `created_at` | 🔶 желательно | время возникновения алерта, UK сохранит |
| `correlation_id` | 🔶 желательно | UK сохранит, см. J1 |
| `infrastructure_type` / `infrastructure_id` | 🔶 опционально | сохраним в raw для отображения, не для логики |
| `metric_id` / `metric_value` / `metric_unit` | 🔶 опционально | сохраним в raw для отображения |
| `building_ids[]` | ❌ **не нужен** | см. ниже |

**Один алерт = одно здание = одна заявка.** Если авария ТП затронула 5 зданий —
шлите **5 отдельных `alert.created`**, каждый со своим `event_id` и одним
`external_id`. Так проще идемпотентность и трассировка. UK **не** будет
разворачивать массив в N заявок.

**A2. Severity.** UK примет словарь `low` / `medium` / `high` / `critical` —
соответствует вашему. 🔶 Маппинг severity→urgency — на стороне UK.

**A3. Категория / urgency.** 🔶 **UK выводит сам** из `type` + `severity`.
Не передавайте `uk_category` / `uk_urgency` — таксономия категорий UK
(`Электрика`, `Сантехника`, ...) принадлежит UK и не должна дублироваться в
`alert_rules`. От вас — стабильный словарь `type`. Пришлите полный перечень
возможных `type` — UK составит маппинг `type → category`.

**A4. Locale.** ✅ `message` — на русском, как сейчас.

## B. События за пределами `alert.created`

**B1.** ✅ Фаза 1 принимает **любое** значение `event` в конверте (валидируется
только подпись + наличие поля). 🔶 Фаза 2 обрабатывает **только `alert.created`**.
`alert.acknowledged/resolved/escalated/deleted` Фаза 2 примет (202), но
проигнорирует, пока не будет отдельно реализована их обработка.

**B2.** 🔶 Рекомендация: статусные апдейты (`acknowledged`/`resolved`) — **через
тот же endpoint** с другим `event`, отдельный endpoint не нужен. Но: интересует
ли UK ваш `resolve` — зависит от модели. Если метрика вернулась в норму до того,
как исполнитель UK закрыл заявку — UK **не** будет авто-закрывать заявку (заявка
живёт по своему циклу: исполнитель → менеджер). `alert.resolved` UK может
сохранить как пометку в заявке, не более. Финал — при дизайне Фазы 2.

## C. Обратная связь (UK → InfraSafe)

**C1.** ✅ Актуально, не меняется. Исходящий канал UK→InfraSafe уже в проде:
`request.created` и `request.status_changed` уходят в `POST /api/webhooks/uk/request`
(см. `api/requests/router.py:265,375`, транзакционный outbox).
⚠️ Но: см. **ARCH-113** в UK-бэклоге — сейчас `request.*` эмитятся **только** при
создании заявки через REST API (дашборд), бот-путь не эмитит. Для Фазы 2
(заявка из вашего алерта создаётся на бэкенде) emit будет настроен явно.

**C2.** 🔶 Фаза 2: UK сохранит связь `event_id` (ваш) ↔ `request_number` (UK) в
служебной таблице маппинга и **вернёт ваш `event_id`** в payload
`request.created`. Полагаться на `alert_id` не нужно — ключ матчинга `event_id`.

## D. Идемпотентность и retry

**D1.** ✅ Фаза 1: тело 409 — `{"detail":"duplicate event"}`, **без**
`request_number` (в Фазе 1 заявка не создаётся). 🔶 Фаза 2: 409 будет возвращать
`request_number` ранее созданной заявки — чтобы вы восстановили маппинг.

**D2.** ✅ Фаза 1: dedup — Redis TTL **600s**. После 600s тот же `event_id`
пройдёт как новый. ⚠️ Плюс: подпись имеет окно 300s — легитимный retry на 11-й
минуте с **исходным `t`** и так упадёт на 401 (stale). Дубль-заявка возможна
только если sender переподпишет с новым `t`. 🔶 **Фаза 2: UK добавит
персистентный dedup** (таблица `webhook_inbox`, ключ `event_id`) — окно станет
бессрочным. Рекомендация sender'у: не ретраить один `event_id` дольше ~10 мин.

**D3.** ✅ Сейчас `Retry-After` UK **не отдаёт** (ни на 429, ни на 503).
Рекомендуемый backoff: экспоненциальный `2s, 4s, 8s, 16s...` cap ~5 мин — это же
использует UK-outbound (`webhook_sender.py:_BACKOFF_DELAYS`). 🔶 Добавление
`Retry-After` на 429/503 — возьмём в Фазу 2, если нужно.

**D4.** ✅ `event_id` — **любая уникальная строка** (`event_id: str`), UUID v4
не обязателен, ограничения длины нет. Рекомендация — UUID v4 для гарантии
уникальности. Используется как Redis-ключ — избегайте пробелов/спецсимволов.

## E. Аутентификация и секреты

**E1.** ✅ **Два направления — два секрета:**
- InfraSafe → UK (этот endpoint): UK verifier читает **`UK_WEBHOOK_SECRET`**
  (+ `UK_WEBHOOK_SECRET_NEXT`). Sender InfraSafe подписывает им же.
- UK → InfraSafe (`/api/webhooks/uk/request`): UK подписывает
  **`INFRASAFE_WEBHOOK_SECRET`** (+ `_NEXT`).
Это разные значения. Не путать.

**E2.** ⚠️ UK принимает `OLD‖NEW` всё время, пока **оба** заданы в `.env`
(`UK_WEBHOOK_SECRET` + `UK_WEBHOOK_SECRET_NEXT`) — verifier перебирает оба.
Окно — на ваше усмотрение, рекомендуем **≥24 ч**. Процедура — integration-plan
§4.4. Конкретное значение `UK_WEBHOOK_SECRET` (ротация после FIX-002) согласуем
отдельно перед go-live.

**E3.** ⚠️ Отдельного staging UK сейчас нет (проект ведёт один разработчик).
Вариант для smoke — поднять Фазу 1 на dev с тестовым секретом и прогнать события
туда (см. L2). Постоянный staging — отдельная инфраструктурная задача.

## F. Endpoint и сеть

**F1.** ✅ Нового inbound endpoint — `POST /api/v2/webhooks/infrasafe/alert`.
Базовый URL — тот же UK API. **`/api/v1/requests` на UK не существует** (см. G).

**F2.** ⚠️ Production: предпочтительно **internal через `uk-network`** (без выхода
в публичный интернет). Внешний путь `https://infrasafe.uz/uk/api/v2/webhooks/infrasafe/alert`
тоже работает. UK ожидает любой из двух — выберите internal, если контейнеры в
одной сети.

**F3.** ✅ IP-allowlist UK **не применяет**. Защита — HMAC-подпись + rate-limit.
mTLS не используется.

**F4.** ✅ TLS терминируется Caddy (prod edge) — TLS 1.2+ (дефолт Caddy),
pinning нет. Для internal-пути (`uk-network`) — обычный HTTP внутри доверенной
docker-сети.

## G. Существующий JWT-путь `/api/v1/requests`

**G1/G2/G3.** ⚠️ **Важно: `POST /api/v1/requests` на UK-стороне не реализован.**
Греп по `uk_management_bot/api/` — такого endpoint нет (есть только `/api/v1/media/*`
— это прокси к media-сервису, не к вам). `UK_SERVICE_USER`/`UK_SERVICE_PASSWORD`
и GET `/requests/counts-by-building` на UK тоже отсутствуют.

Похоже, JWT-путь из старого integration-plan **никогда не доводился** —
inbound-канал реализуется впервые именно как FIX-007 webhook. Следствия:
- **G1** — выпиливать нечего, sunset не нужен.
- **G2** — параллельный режим не нужен; новый HMAC-webhook = единственный
  inbound-канал.
- **G3** — `UK_SERVICE_USER/PASSWORD` для UK не нужны. Если вам нужны GET-данные
  от UK (counts-by-building и т.п.) — это **отдельный запрос**, такого API
  сейчас нет, спроектируем при необходимости.

⚠️ Если у вас уже есть рабочая отправка в какой-то UK endpoint — сообщите,
сверим: возможно, это шло в InfraSafe-собственный `/api/webhooks/uk/*`.

## H. Rate-limit и backpressure

**H1.** ✅ **Fixed-window**, 60/мин, per source-IP (slowapi, `strategy="fixed-window"`).
Не sliding, не token-bucket.

**H2.** ⚠️ 30 алертов/сек — **упрётся в 429** (лимит 60/мин ≈ 1/сек sustained).
Безопасный sustained rate сейчас — **≤60/мин**. Burst сверх лимита в текущем окне
получит 429.

**H3.** ✅ Batch-endpoint сейчас **нет**. 🔶 Если шторм-сценарий (отказ ТП → залп
алертов) реален — это весомый аргумент за batch-endpoint в Фазе 2
(`POST /infrasafe/alert/batch`, массив событий, одна подпись). Подтвердите
ожидаемый пик — заложим.

**H4.** ⚠️ Нужна ваша оценка типовой и пиковой нагрузки (алертов/час и пик при
аварии). Если пик реально >60/мин — поднимем лимит для вашего IP или сделаем
batch (H3). Без ваших цифр дефолт 60/мин.

## I. Ответы и формат ошибок

**I1.** ✅ Фаза 1: ошибки — **дефолтный FastAPI-конверт** `{"detail": "<текст>"}`,
без структурированного `fields[]`. 422 → `{"detail":"invalid payload schema"}`.
🔶 Если вам нужен машиночитаемый формат с `fields[]` — скажите, добавим в Фазе 2.

**I2.** ✅ 401 **различает причину** в `detail`: `signature no_header` /
`signature bad_format` / `signature stale` / `signature no_match`. Удобно для
отладки sender'а.

**I3.** ✅ 503 → `{"detail":"webhook receiver not configured"}`, **без**
`Retry-After`. 503 означает, что на UK не задан `UK_WEBHOOK_SECRET` — это
операторская проблема UK, ретрай поможет только после её устранения.

## J. Observability

**J1.** ✅ Фаза 1 `x-correlation-id` **не обрабатывает** и не возвращает.
🔶 Рекомендация: кладите `correlation_id` **внутрь `alert`** (см. A1) — UK
сохранит. Поддержку header `x-correlation-id` с эхом в response добавим в Фазе 2,
если нужно.

**J2.** ✅ Фаза 1 — аудит только в **структурированных логах** (`event_id`,
source_ip, outcome, причина). Персистентного журнала входящих вебхуков пока нет.
🔶 Фаза 2 — таблица `webhook_inbox` (durable audit + idempotency, см. D2);
тогда появится journal для разбора инцидентов.

**J3.** ⚠️ Авто-уведомления вам при стабильных провалах подписи/схемы сейчас
**нет**. UK-сторона видит это в логах. Настройка алертинга (например, при росте
401/422) — отдельная OPS-задача; на go-live согласуем канал связи для инцидентов.

## K. Время и NTP

**K1.** ⚠️ UK-хосты синхронизируются штатным NTP ОС (контейнеры наследуют время
docker-host). Систематического сдвига быть не должно. Если увидите дрейф —
сообщите, окно 300s имеет запас.

**K2.** ✅ UK сверяет **только `t` из header-подписи** (окно 300s). Поле
`timestamp` внутри payload **не проверяется** — это информационная метка времени
события, хранится как есть. Подписанный `t` и `timestamp` могут отличаться (UK
не требует их равенства).

## L. Тестовые fixture / sanity

**L1. ✅ Reference-вектор для pin-теста** (ваш sender должен сгенерить ровно эту
подпись):

```
secret  : uk_webhook_shared_secret_DEMO
t       : 1747900800
body    : {"event_id":"11111111-1111-4111-8111-111111111111","event":"alert.created","timestamp":"2026-05-22T12:00:00Z","alert":{"severity":"high","message":"test"}}
message подписи : "1747900800." + body   (т.е. "1747900800.{...}")
v1 (HMAC-SHA256 hex) : 6fe9e4327b7e9c9e22e49442cc376650358ca7894b1102615274e0ba9d47a1dc
header  : x-webhook-signature: t=1747900800,v1=6fe9e4327b7e9c9e22e49442cc376650358ca7894b1102615274e0ba9d47a1dc
```

`body` подписывается и отправляется **байт-в-байт**. Этот `t` в проде даст 401
(stale) — для smoke используйте свежий `t`, вектор нужен только для проверки
симметрии алгоритма.

**L2.** 🔶 Да — после готовности вашего sender'а UK поднимет Фазу 1 на dev с
тестовым секретом, прогоните ~10 событий, ждём 202. Phase 1 endpoint для этого
подходит (он валидирует весь envelope, просто не создаёт заявку).

**L3.** Negative-кейсы для покрытия — UK уже тестирует их у себя (10/10 зелёных).
Со стороны sender'а проверьте: (a) нет header → 401; (b) подпись чужим
секретом → 401; (c) `t` старше 300s → 401; (d) тело без `event_id` → 422;
(e) тот же `event_id` дважды → 409.

## M. Phase 2 timeline и зависимости

**M1/M2.** ⚠️ Требует решения UK (ведёт один разработчик). Фаза 1 — код готов,
10/10 тестов зелёные, ждёт деплоя `uk-management-api`. Фаза 2 — отдельный
ticket, оценка после фиксации схемы `alert` (раздел A) и решения echo-loop с
ARCH-113. Конкретные даты — сообщим после планирования.

**M3.** ⚠️ Важно: к моменту приёма `alert.created` по зданию UK должен **уже
иметь это здание** (Фаза 2 резолвит `external_id` → building). Здания
синхронизируются UK→InfraSafe (`building.created` webhook + hourly
reconciliation). Перед первым алертом по новому зданию убедитесь, что
`building.created` доехал. Если UK не найдёт здание по `external_id` — Фаза 2
вернёт ошибку (код согласуем — вероятно 422 или 409).

**M4. Backfill.** 🔶 Рекомендация: при downtime UK **не** слать накопленное
скопом (упрётесь в 60/мин + риск дублей). Дренируйте очередь плавно ≤60/мин.
Идемпотентность по `event_id` (после персистентного dedup Фазы 2) защитит от
дублей. До Фазы 2 — старые алерты лучше дропать, UK Фаза 1 их всё равно не
обрабатывает.
